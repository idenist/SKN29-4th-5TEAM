# 프론트엔드 구현 현황

작성 기준: 현재 저장소 코드 기준

현재 문서 버전: **ver 2.7**

---

## 1. 프로젝트 개요

### 프로젝트명

**이젠, 안심**

### 목적

청년 정책 정보를 한곳에서 검색하고, AI 상담, 커뮤니티, 뉴스, 영상, 알림, 마이페이지 기능을 통해 사용자가 필요한 정책과 정보를 쉽게 확인할 수 있도록 만든 서비스입니다.

### 사용자 주요 흐름

- 홈 화면에서 추천 정책, 정책 뉴스, 인기 검색어를 확인합니다.
- 정책명, 설명, 나이, 분야, 지역, 상태, 소득조건을 조합해 정책을 검색합니다.
- 검색 조건이 없을 때는 기본 20개 정책 목록을 자동으로 보여주지 않고, 검색 결과 없음 상태를 표시합니다.
- 정책 상세 화면에서 지원 대상, 신청 방법, 기간, 링크 등 상세 정보를 확인합니다.
- 로그인 사용자는 정책을 스크랩하고 마이페이지에서 관리합니다.
- AI 챗봇에 청년 정책 관련 질문을 입력하고 추천 정책을 확인합니다.
- 커뮤니티에서 게시글을 작성, 조회, 수정, 삭제합니다.
- 회원가입 시 이메일 인증번호 확인 후 가입합니다.
- 로그인 화면에서 인증했던 이메일로 비밀번호 찾기 및 재설정을 진행합니다.
- 마이페이지에서 비밀번호를 변경할 수 있습니다.
- 로그인 시 비밀번호를 일정 횟수 이상 틀리면 이메일 인증 기반 비밀번호 재설정으로 유도합니다.
- 헤더 알림 버튼에서 확인하지 않은 알림 수를 확인하고, 버튼 클릭 시 알림 목록을 확장해 볼 수 있습니다.

## 2. ver 2.7 개선사항 요약

### 정책 검색 / 필터 API 연결

- Header 우측 버전 배지를 `ver 2.7`로 변경했습니다.
- 정책 검색 결과 카드를 첨부 시안 기준의 가로형 리스트 카드로 개편했습니다.
  - 왼쪽 순번 배지와 교육/정책 아이콘 영역을 추가했습니다.
  - 중앙에는 정책명, 상태/분야 배지, 요약, `신청기간`, `대상연령`, `지원내용`을 간결하게 표시합니다.
  - 오른쪽에는 `신청 가능` 또는 `신청 링크 없음`, `출처 보기` 버튼을 배치했습니다.
- 정책 검색 결과 카드의 UX 여백을 재조정했습니다.
  - 카드 간격, 카드 최소 높이, 좌측 아이콘 영역, 본문 정보 영역, 우측 버튼 영역의 상하 여백을 조정했습니다.
  - 긴 제목/요약과 하단 `신청기간`, `대상연령`, `지원내용`이 한 카드 안에서 눌려 보이지 않도록 높이와 간격을 확보했습니다.
- 정책 검색 목록 카드에서 신청 URL과 출처 URL을 사용할 수 있도록 `application_url`, `source_url`, `source_url_2` 매핑을 보강했습니다.
- 정책 검색 필터가 실제 DB 구조와 맞도록 API 요청/응답 흐름을 정리했습니다.
  - 프론트는 분야 선택값을 `category=취업`, `category=교육`, `category=창업`처럼 백엔드로 전달합니다.
  - 기존 프론트 후처리 필터가 API 결과를 다시 잘못 걸러 0건으로 만드는 문제를 제거했습니다.
  - 백엔드는 `category` 값을 DB의 `domain`, `source_category` 구조에 맞게 해석합니다.
  - `취업`은 `domain`의 `취업` 또는 `일자리`, `교육`은 `training` 또는 `교육/education`, `창업`은 `startup_notice` 또는 `창업/startup` 기준으로 조회합니다.
  - `region=전체`이 들어와도 백엔드에서 지역 필터를 적용하지 않도록 방어했습니다.
- 지역 필터는 광역 코드뿐 아니라 하위 시군구 코드가 저장된 데이터도 검색되도록 백엔드 지역 필터 조건을 보강했습니다.
- 정책 검색 페이지네이션을 서버 기준으로 수정했습니다.
  - 검색/필터는 전체 정책 DB를 대상으로 수행합니다.
  - 백엔드는 전체 검색 결과 수 `count`와 현재 페이지 데이터 `results`를 함께 반환합니다.
  - 프론트는 `limit=10`, `offset=(page - 1) * 10`으로 요청해 검색 결과를 10개 단위로 페이지 이동합니다.
  - 화면에는 `총 n개 정책`과 `x-y번째 결과 표시 중` 문구를 표시해 전체 검색 결과와 현재 페이지 범위를 구분합니다.
  - 백엔드 정렬을 `-info_score`, `item_id` 기준으로 고정해 페이지 이동 시 중복 노출을 방지했습니다.

### 지역 / 날짜 표시 정규화

- 정책 추천, 정책 검색, 정책 상세에서 지역 코드가 숫자로 노출되는 문제를 추가 수정했습니다.
  - `location`, `region_codes`, `region` 어디에 숫자 코드가 들어와도 같은 지역 변환 로직을 사용합니다.
  - 예: `44130` -> `충남 천안시`, `51130` -> `강원 원주시`, 부산 하위 지역 코드 묶음 -> `부산`
- 날짜 표시를 전역적으로 `YYYY-MM-DD` 형식으로 통일했습니다.
  - `20260301` -> `2026-03-01`
  - `2026.3.1` -> `2026-03-01`
  - `20260301 ~ 20261130` -> `2026-03-01 ~ 2026-11-30`
- 날짜 포맷 공통 유틸 `frontend/src/utils/dateFormat.js`를 추가했습니다.
- 정책, 뉴스, 커뮤니티, 알림, 마이페이지, 채팅 메시지 날짜 표시가 같은 날짜 포맷 유틸을 사용하도록 정리했습니다.

### 교육훈련 데이터 매핑

- 교육훈련(`source_category=training`) 데이터의 대상자 정보는 `trainTarget`을 우선 사용하도록 백엔드 정책 import/serializer 매핑을 보강했습니다.
- 기존 `realMan`, `courseMan` 기반 데이터도 fallback으로 유지하되, API 응답의 `participation_target`은 `trainTarget`을 우선 반환합니다.
- 프론트에서는 기존 `participation_target` 기반 표시를 유지하므로, 백엔드 매핑 변경 후 카드/상세의 대상 연령 및 신청 대상 표시가 자동 반영됩니다.

### 검증

- `npm.cmd run build` 통과를 확인했습니다.
- `python manage.py check` 통과를 확인했습니다.
- 정책 API 직접 호출로 `category=취업`, `category=교육`, `category=창업`, `category=취업&age=25` 조건에서 결과가 반환되는 것을 확인했습니다.
- 백엔드 로컬 서버를 재시작하고 `/api/policies/?category=취업&limit=5` 응답 `200`을 확인했습니다.
- 정책 API 페이지네이션 검증:
  - `/api/policies/?category=취업&limit=10&offset=0` 기준 전체 `count=1044`, 현재 페이지 `results=10` 확인
  - `/api/policies/?category=취업&limit=10&offset=10` 기준 다음 페이지 `results=10` 확인
  - 1페이지와 2페이지 사이 `item_id` 중복 `0개` 확인

## 3. ver 2.6 개선사항 요약

### 인증 / 세션

- API 응답이 401 Unauthorized일 때 로그인 토큰을 자동 제거하고 전역 세션 만료 안내 배너를 표시하도록 연결했습니다.
- 세션 만료 안내는 루트 레이아웃에서 표시되며 `로그인` 버튼을 눌러 `/login` 페이지로 이동할 수 있습니다.
- Header 우측 버전 배지를 `ver 2.6`으로 변경했습니다.

### 정책 검색

- 정책 검색 필터가 실제 API 조건으로 전달되도록 `domain`, `income_condition`, `deadline_status`, `limit` 파라미터를 보강했습니다.
- 백엔드 정책 목록 API도 `domain`, `income_condition` 필터를 처리하도록 보강했습니다.
- 정책 검색 결과 카드의 기본 정보 박스 높이와 그리드 정렬을 일정하게 맞췄습니다.
- 조건 입력 팁 상자의 여백과 패딩을 줄여 필터 카드와 더 자연스럽게 어울리도록 정리했습니다.
- 정책 카드와 상세 화면에서 지역 코드(`51130` 등)가 그대로 노출되지 않도록 `location` 우선 표시 및 `region_codes` 지역명 변환을 적용했습니다.
- 정책 검색 조건 필터의 나이 숫자 입력 스피너가 hover 여부와 관계없이 항상 보이도록 조정했습니다.

### Home / MyPage UI

- 홈 추천 정책 미리보기 카드 높이를 일정하게 맞추고 긴 설명은 3줄 기준으로 정리했습니다.
- 마이페이지 프로필, 요약, 정책 목록, 알림 카드의 최소 높이와 내부 정렬을 맞춰 위아래 크기 차이를 줄였습니다.
- 마이페이지 프로필 카드에서는 `프로필 수정` 버튼만 표시하고, `비밀번호 변경` 버튼은 프로필 수정 화면 내부로 이동했습니다.
- 프로필 수정 화면의 개발자용 안내 문구(`이름/닉네임은 현재 백엔드 프로필 수정 API에서 지원하지 않습니다.`)를 제거했습니다.
- 프로필 수정 화면의 나이 숫자 입력 스피너가 hover 여부와 관계없이 항상 보이도록 조정했습니다.

### 뉴스

- 뉴스 페이지 첫 진입 시 특정 키워드가 기본 선택된 것처럼 보이지 않도록 기본 active 상태를 제거했습니다.
- 기본 뉴스 목록은 전체성 키워드인 `청년정책` 기준으로 불러오고, 사용자가 키워드를 선택했을 때만 해당 키워드 버튼을 선택 상태로 표시합니다.

### 커뮤니티

- 댓글 삭제 API를 프론트에 연결하고, 본인이 작성한 댓글 오른쪽 끝에 `삭제` 버튼이 보이도록 추가했습니다.
- 댓글 삭제 후 상세 화면의 댓글 목록과 댓글 수가 즉시 갱신되도록 처리했습니다.
- 댓글 작성 후 댓글 수는 기존 댓글 수 기준으로 증가하도록 보정했습니다.
- 커뮤니티 목록 테이블에 조회수, 좋아요 수 옆으로 댓글 수 컬럼을 추가했습니다.
- 게시글에 좋아요 또는 댓글이 달리면 글 작성자에게 알림이 생성되도록 백엔드 커뮤니티 API를 보강했습니다.
- 좋아요 API 응답 포맷을 프론트 상태 갱신에 맞게 `post_id`, `is_liked`, `likes`, `like_count` 기준으로 정리했습니다.
- 중복 좋아요 모델 마이그레이션 충돌을 정리해 배포 시 migration 충돌 가능성을 줄였습니다.

### AWS / LLM 배포 점검

- 프론트 빌드, Django `check`, Django `makemigrations --check --dry-run`, 로컬 `migrate`를 통과했습니다.
- 커뮤니티 좋아요, 댓글 작성, 댓글 삭제, 알림 생성 흐름을 Django APIClient로 확인했습니다.
- 배포 전 확인 필요 사항:
  - 운영 `.env`의 `DJANGO_DEBUG`는 `False`로 변경해야 합니다.
  - EC2 도메인/IP를 `DJANGO_ALLOWED_HOSTS`, `CORS_ALLOWED_ORIGINS`, `CSRF_TRUSTED_ORIGINS`에 맞춰 넣어야 합니다.
  - 프론트를 HTTPS로 서비스할 경우 `VITE_API_BASE_URL`도 HTTPS API 주소로 맞춰야 mixed content 문제가 없습니다.
  - LLM/RAG 운영에는 `OPENAI_API_KEY`, `CHROMA_PERSIST_DIR`, `CHROMA_COLLECTION_NAME`이 필요합니다.
  - 외부 검색 fallback을 사용할 경우 `TAVILY_API_KEY` 또는 대체 검색 API 키가 필요합니다.

## 5. ver 2.4 개선사항 요약

### Home / AI 챗봇 흐름

- 홈 히어로 검색창을 정책 검색 진입점이 아니라 `AI 안쉼 챗봇` 질문 진입점으로 확정했습니다.
- 홈 검색창, 추천 질문, 인기 검색어 클릭 시 모두 `/chat?message=...`로 이동하도록 연결했습니다.
- 챗봇 페이지는 URL의 `message` 값을 읽어 최초 진입 시 해당 질문을 자동 전송합니다.
- 홈 히어로 배지 문구에서 `New! 이젠`을 제거하고 `AI 맞춤형 추천 서비스`만 표시하도록 정리했습니다.
- 정책 검색은 홈 키워드 검색 진입보다 나이, 분야, 지역, 상태, 소득조건 기반 조건 필터 중심으로 사용하는 흐름으로 분리했습니다.

### 커뮤니티

- 커뮤니티 상세 화면에 좋아요 버튼을 추가하고 로그인 사용자가 좋아요 등록/취소를 할 수 있도록 API를 연결했습니다.

## 10. API 연동 현황

### 인증

| 기능 | 프론트 서비스 | 비고 |
| --- | --- | --- |
| 로그인 | `authApi.login` | 실패 제한 상태 코드 처리 |

## 14. 백엔드 협업 필요 사항

- 커뮤니티 게시글 응답에 `author_name`, `author_id`, `category`, `category_label`이 안정적으로 포함되어야 합니다.
- 커뮤니티 `CommunityPost.category` 필드와 마이그레이션이 배포 DB에 적용되어야 합니다.
- 회원가입 이메일 인증번호 발송/검증 API가 운영 SMTP 설정과 연결되어야 합니다.
- 비밀번호 재설정용 인증번호 발송/검증 API가 운영 SMTP 설정과 연결되어야 합니다.
- 로그인 실패 제한은 운영 환경에서 캐시 또는 Redis 기반으로 관리하는 것이 권장됩니다.
- 알림 API는 미확인 알림 수, 읽음 처리, 전체 읽음, 삭제 기능을 유지해야 합니다.
- 정책 검색 나이 조건은 프론트에서 `age` 파라미터로 전달하므로 백엔드 필터 기준과 응답 범위를 확인해야 합니다.
