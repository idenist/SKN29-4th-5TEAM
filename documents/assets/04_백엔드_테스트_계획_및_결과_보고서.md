### 2.1 환경 관련 이슈 (버그 아님, 환경설정 주의사항)

`.env`의 `DB_HOST` 값이 `db`로 설정되어 있는데, 이는 **Docker 컨테이너 환경 전용 값**이다(docker-compose 네트워크 안에서 서비스명 `db`로 접속). 로컬 터미널에서 `python manage.py test`를 직접 실행하면 로컬 OS가 `db`라는 호스트명을 해석하지 못해 다음 에러가 발생한다.

```
psycopg2.OperationalError: could not translate host name "db" to address
```

**대응**: 로컬에서 테스트할 때는 아래처럼 환경변수를 임시로 덮어써서 실행한다.

```bash
DB_HOST=localhost python manage.py test tests
```

**팀 공유 필요**: 이 환경변수 문제는 다른 팀원이 로컬에서 테스트를 돌릴 때도 동일하게 발생할 수 있으므로, README 또는 팀 공유 문서에 안내가 필요하다.

## 4. 발견 이슈 및 수정 이력

### 4.1 [기능 누락] 정책 검색 age 필터 미구현

- **발견 경위**: TC-POLICY-03 테스트 코드 작성 중, `Policy` 모델에 연령 관련 필드(`age_min`/`age_max`)가 존재하지 않고, `PolicyListView.get_queryset()`에도 `keyword`/`region`/`source_category` 필터만 구현되어 있고 `age` 필터가 아예 없음을 확인.
- **원인 분석**: 지침서상 목표 기능이었으나 구현 우선순위에서 누락된 것으로 확인. 평가계획서(실제 채점 기준) 자체에는 age 필터가 명시적 배점 항목이 아님을 별도 확인.
- **팀 논의**: "전체 테스트 스위트에 계속 들어갈 항목이니, 나중에 안 쓰더라도 지금 구현해두는 것이 낫다"는 판단 하에 신규 구현하기로 결정.
- **조치**:
  1. `Policy` 모델에 `age_min`, `age_max` 필드 추가 (nullable, 기존 데이터 영향 없음)
  2. Migration 생성 및 적용 (`0002_policy_age_max_policy_age_min.py`)
  3. `PolicyListView.get_queryset()`에 age 필터 로직 추가 (`age_min`/`age_max`가 null인 정책은 "연령 제한 없음"으로 간주하여 항상 포함)
- **재검증 결과**: 신규 테스트 `PolicyAgeFilterTests` (연령 범위 내 포함, 범위 밖 제외 두 케이스) 통과 확인.
- **후속 조치 필요**: `age_min`/`age_max`는 현재 전부 `NULL` 상태. 정승의 데이터 적재 시 이 필드를 채울지, 윤승혁의 RAG 추천 로직(지침서 6.4 "사용자 나이가 정책의 age_min/age_max 범위에 들어가는지 확인")에서 이 필드를 참조할지 팀 협의 필요.

### 4.2 [버그] age 필터 구현 중 `UnboundLocalError` 발생

- **발견 경위**: 4.1 조치 이후 테스트 실행 시 아래 에러로 7개 테스트 ERROR 발생.
  ```
  UnboundLocalError: cannot access local variable 'age' where it is not associated with a value
    File "apps/policies/views.py", line 57, in get_queryset
      if age:
  ```
- **원인**: `get_queryset()` 내에서 `age = self.request.query_params.get("age")` 로 변수를 초기화하는 줄이 누락된 상태로 `if age:` 를 먼저 참조함. Python은 함수 내에서 이후에 `age = int(age)`로 재할당하는 코드가 있으면 해당 변수를 함수 전체의 지역변수로 취급하는데, 초기화 없이 먼저 읽으려 해서 발생.
- **재현 방법**: `GET /api/policies/?keyword=월세` 등 **age 파라미터 없이** 정책 목록 API를 호출하면 무조건 500 에러 발생 (age 유무와 무관하게 `get_queryset()` 자체가 항상 이 코드를 거치기 때문).
- **영향 범위**: `PolicyListView`를 사용하는 모든 요청 (목록 조회, keyword/region/source_category 검색 전부 포함) — **정책 검색 기능 전체가 마비되는 심각도 높은 버그**였음.
- **수정**: `age = self.request.query_params.get("age")` 초기화 줄 추가.
- **재검증 결과**: 수정 후 전체 50개 테스트 통과 확인. 이후 공통 예외 처리 작업을 거쳐 최종 53개 전체 통과.
- **비고**: 자동 테스트가 없었다면 이 버그는 배포 후 정책 검색 API 전체 장애로 이어질 수 있었던 사안. 테스트 코드 작성의 실효성을 보여주는 대표 사례.

### 4.3 [테스트 코드 초기 가정 오류] Policy 모델 필드명 불일치

- **발견 경위**: 테스트 코드를 실제 코드 확인 없이 지침서 예시 필드명(`region`, `age_min`, `age_max`, `is_active`) 기준으로 먼저 작성 → 실행 시 `TypeError: Policy() got unexpected keyword arguments` 발생.
- **원인**: 실제 `Policy` 모델은 `region` 대신 `region_codes`(JSON 리스트), `is_active` 필드 자체가 없는 등 지침서 예시와 실제 구현이 다름.
- **조치**: 실제 `models.py` 원본 확인 후 테스트 코드의 fixture 함수(`create_policy()`)를 실제 필드명 기준으로 전면 수정.
- **비고**: 코드 버그가 아니라 테스트 작성 과정에서의 정상적인 보정 과정. 다만 이 과정에서 팀 문서(지침서)와 실제 구현 간 괴리가 존재함을 확인했으므로, 지침서의 Policy 모델 필드 예시 부분은 실제 구현 기준으로 업데이트가 필요함을 별도 기록.

### 4.4 [테스트 코드 초기 가정 오류] 스크랩 API 요청 필드명 불일치

- **발견 경위**: 스크랩 생성 요청을 `{"item_id": ...}`로 보내도록 테스트를 작성했으나, 실제 `ScrapSerializer`는 `policy`(FK, PK가 `item_id`인 CharField) 필드를 받는 구조임을 `serializers.py` 확인 후 발견.
- **조치**: 테스트 요청 payload를 `{"policy": item_id}`로 수정.
- **비고**: 프론트엔드(송민지) 연동 시 실제로 이 필드명(`policy`)으로 요청을 보내야 하므로, API 명세 문서에 정확히 반영되어 있는지 별도 확인 필요.

### 4.5 [테스트 코드 초기 가정 오류] 프로필 이미지 업로드 관련 3건

- **응답 코드 불일치**: 업로드 성공 시 응답 코드가 200이 아니라 **201 (CREATED)** 임을 `views.py` 확인 후 발견 → 테스트 기대값 수정.
- **가짜 이미지 바이트 검증 실패**: `serializers.ImageField()`가 Pillow로 실제 이미지 여부를 검증하는데, 테스트에서 `b"fake-image-bytes"` 같은 가짜 바이트를 사용해 항상 400이 발생 → Pillow로 실제 유효한 이미지(1x1 JPEG)를 생성하는 헬퍼 함수로 교체.
- **mock 대상 누락**: 업로드 성공 로직 내부에서 `delete_profile_image()`(기존 이미지 삭제)도 함께 호출되는데, 이를 mock 처리하지 않아 실제 S3 호출을 시도하려는 문제 발견 → `upload_profile_image`, `delete_profile_image` 둘 다 mock 처리.
- **비고**: 셋 다 코드 버그가 아니라 테스트 작성 시 실제 로직을 정확히 파악하지 못해 발생한 것으로, `views.py`/`serializers.py` 원본 확인 후 모두 해결.

## 5. 개선 이력 (기능 추가) — 공통 예외 처리 유틸

지침서 9.2 "가능하면 해야 할 일" 마지막 항목으로, 테스트 코드 작성과는 별개로 아래 개선을 진행함.

### 5.1 배경

기존에는 `policies`, `uploads`, `community`, `notifications` 4개 앱이 각자 파일 상단에 동일한 `success_response()`/`error_response()` 함수를 복붙해서 사용 중이었음. 응답 형식을 바꾸려면 4개 파일을 전부 수정해야 하는 구조였고, DRF가 처리하지 못하는 예상 밖 예외(DB 연결 오류, 코드 버그 등)가 발생하면 Django 기본 500 HTML 에러 페이지가 그대로 API 응답으로 노출되는 문제가 있었음.

### 5.2 조치

1. `apps/common/responses.py` 신설 — `success_response`/`error_response`를 한 곳으로 통합
2. `apps/common/exceptions.py` 신설 — DRF `EXCEPTION_HANDLER`를 커스터마이징하여, DRF가 처리하는 예외(ValidationError, 404, 인증 실패 등)와 처리하지 못하는 예외(500) 모두 동일한 응답 형식(`{"success": false, "data": null, "message": ..., "error": ...}`)으로 통일. 처리되지 않은 예외는 서버 로그에 자동 기록되도록 `logger.exception()` 처리.
3. `settings.py`의 `REST_FRAMEWORK` 딕셔너리에 `"EXCEPTION_HANDLER": "apps.common.exceptions.custom_exception_handler"` 추가
4. `policies`, `uploads`, `community`, `notifications` 4개 앱의 `views.py`에서 로컬 함수 정의 제거, `apps.common.responses`에서 import하도록 수정

### 5.3 검증

`test_exception_handling.py` 신규 작성:
- 404 응답(`test_404_response_uses_common_format`)
- 회원가입 검증 실패(`test_validation_error_uses_common_format`)
- 인증 필요 API 미인증 접근(`test_auth_required_error_uses_common_format`)

3개 케이스 모두 응답 바디에 `success`, `data`, `message`, `error` 키가 일관되게 포함됨을 확인. 전체 회귀 테스트(53개) 통과로 기존 API 동작에 영향 없음을 확인.
