# 🏛️ 청년 정책 맞춤형 추천 플랫폼 시스템 요구사항 명세서 (SRS)

본 문서는 SKN 29기 4차 프로젝트 5팀이 개발하는 '청년 정책 맞춤형 추천 플랫폼'의 핵심 기능 구동을 위한 컴포넌트별 상세 요구사항 및 인터페이스, 데이터베이스(DB) 엔티티 설계를 규정하는 전문 기능 명세서입니다.

---

## 1. 시스템 아키텍처 및 데이터 흐름 개요

본 플랫폼은 비동기 사용자 반응형 프론트엔드(React/Vite)와 엄격한 유효성 검증 및 데이터 제어가 가능한 백엔드(Django REST Framework) 구조를 가집니다. 외부 오픈 API(Naver Search API) 및 지능형 RAG 워크플로우(LangGraph, OpenAI, Chroma Vector DB)와 유기적으로 연동되어 작동합니다.

```
[Frontend (Vite Proxy)] ──(API Requests)──> [Backend (DRF / Gunicorn / Nginx)]
                                                      │
         ┌────────────────────┬───────────────────────┼──────────────────────┐
         ▼                    ▼                       ▼                      ▼
 [Naver News API]   [PostgreSQL (AWS RDS)]   [Email SMTP Server]   [LangGraph RAG Engine]
 (뉴스 데이터 프록시)  (도메인/활동 트래킹)      (인증 코드 발송)        (벡터/하이브리드 검색)
```

## 2. 컴포넌트별 기능 요구사항 (Functional Requirements)

### 2.1. 외부 API 프록시 기반 뉴스 시스템 (`COMP-NEWS`)

#### 2.1.1. 프론트엔드 뉴스 페이지 컴포넌트 설계 요구사항
* **REQ-FE-NEWS-01 (키워드 기반 필터링 탭 바):**
    * 시스템은 상단에 8대 핵심 청년 정책 검색 키워드로 구성된 고정형 탭 바를 제공해야 한다.
    * 정의된 키워드: `['청년정책', '청년수당', '청년월세지원', '청년도약계좌', '내일배움카드', '취업장려금', '창업지원', '주거지원']`
    * 사용자가 특정 키워드를 클릭하면 즉시 상태(State)가 변경되며 해당 키워드를 조건으로 뉴스 목록을 API에 재요청한다. 초기 진입 시의 기본값은 `'청년정책'`으로 설정한다.
* **REQ-FE-NEWS-02 (가이드 배너 및 메인 피드 배치 규칙):**
    * 최상단 타이틀 영역 하단에 블루 그라데이션 톤 배너(배경색: `#eff6ff`, 보더: `1px solid #bfdbfe`, 폰트색: `#1e40af`)를 배치하여 정보 확인 유의사항 안내 문구를 상시 노출한다.
    * 조회된 뉴스 리스트 중 첫 번째 인덱스의 데이터는 '오늘의 주요 브리핑' 카드 세그먼트로 격상하여 렌더링한다. 주요 브리핑 카드는 연한 하늘색 배경색(`#f0f7ff`)과 강조 보더(`1px solid #bfdbfe`)를 적용하여 시각적 가독성을 높인다.
    * 나머지 뉴스 항목(인덱스 1 이후)은 흰색 바탕(`#ffffff`)의 정형 가로 리스트형 카드로 하단에 종속 렌더링한다. 왼쪽 영역에는 고정 문서 아이콘 SVG를 배치하여 균일한 레이아웃을 유지한다.
* **REQ-FE-NEWS-03 (컴포넌트 결합도 및 텍스트 예외 처리):**
    * `NewsPage` 구역은 데이터를 하위 `NewsList` 및 `NewsCard` 컴포넌트로 전파할 때 다음 인터페이스 규격을 상시 만족해야 한다: `title`, `summary`, `source`, `publishedAt`, `category`, `tags`, `url`.
    * 모든 뉴스 카드의 제목 및 "원문 보기 ↗" 요소 클릭 시 브라우저 보안 제약을 방지하고 이탈률을 제어하기 위해 `target="_blank"` 및 `rel="noopener noreferrer"` 속성이 적용된 외부 하이퍼링크로 처리되어야 한다.
    * 비동기 패치 중에는 `Spinner` 컴포넌트를 호출하고, 예외 발생 시 전면 에러 레이아웃(`"뉴스 정보를 불러오지 못했습니다. 잠시 후 다시 시도해주세요."`)을 렌더링해야 한다.

#### 2.1.2. 백엔드 뉴스 프록시 API 요구사항
* **REQ-BE-NEWS-01 (CORS 회피 및 서버사이드 API 바인딩):**
    * 브라우저에서 Naver 실시간 속보 API를 직접 호출할 때 발생하는 CORS 제약을 원천 차단하기 위해, 웹 서버 Nginx 및 DRF 단에서 우회 전용 엔드포인트를 구현해야 한다.
    * 백엔드는 프론트엔드가 요청한 `keyword` 파라미터를 수신하여 외부 엔드포인트(`https://openapi.naver.com/v1/search/news.json`)로 내부 HTTP GET 요청을 위임 처리한다. 이때 서버 환경 변수에 격리 보관된 `X-Naver-Client-Id` 및 `X-Naver-Client-Secret` 헤더 값을 바인딩하여 서명해야 한다.
* **REQ-BE-NEWS-02 (파라미터 정규화 및 예외 대응):**
    * 조회 개수를 제어하는 `display` 파라미터 유입 시 정수 변환 유효성 검사를 거쳐야 하며, 범위를 초과하는 악성/비정상 유입 시 최소 1개, 최대 20개의 안전한 경계 범위(Bound) 내로 강제 산술 정규화 처리를 수행해야 한다.
    * 외부 네이버 API 장애 또는 네트워크 타임아웃 발생 시, 시스템은 무중단 서빙을 위해 구조적 예외 처리를 수행하고 프론트엔드에 `HTTP_502_BAD_GATEWAY` 표준 에러 페이로드를 안정적으로 반환해야 한다.

### 2.2. 이메일 SMTP 연동 보안 인증 시스템 (`COMP-AUTH`)

#### 2.2.1. 회원가입(Signup) 보안 파이프라인 요구사항
```
[회원가입 메일 발송] ──> [인증 코드 확인] ──> [비밀번호 유효성 검증] ──> [최종 가입 완료]
  (인증 모델 레코드 생성)     (is_verified=True)      (Django 내장 검증기 작동)     (인증 코드 사용 무효화)
```
* **REQ-AUTH-SIGN-01 (이메일 중복 및 인증 코드 난수 발급 로직):**
    * 인증 코드 요청(`POST /api/auth/signup/email/send/`) 수신 시, 시스템은 우선 `User` 테이블에 동일한 이메일 계정이 존재하는지 조회하여 기 가입된 이메일일 경우 즉시 `ValidationError("이미 가입된 이메일입니다.")`를 발생시켜 프로세스를 중단해야 한다.
    * 검증 통과 시 `random.randint(0, 999999)` 엔진을 구동하여 6자리의 정수형 난수 코드를 생성하되, 자릿수 유실을 방지하기 위해 스트링 포맷팅 자릿수 패딩 조치(`:06d`)를 필수 적용해야 한다.
* **REQ-AUTH-SIGN-02 (동일 목적 레코드 무효화 및 만료 통제):**
    * 새로운 인증 코드를 발급하기 직전, 시스템은 데이터베이스 내 동일 이메일(`email`)과 동일 목적(`purpose='signup'`)을 가진 기존의 미사용(`is_used=False`) 코드 레코드들을 일괄 조회하여 전건 `is_used=True` 상태로 변환(무효화)함으로써 보안 위협을 예방해야 한다.
    * 신규 레코드 저장 시 만료 타임스탬프(`expires_at`)를 동적으로 계산하여 주입하고, Django 내장 `send_mail` 엔진을 통해 `"[이젠, 안쉼] 회원가입 인증번호"` 제목의 메일을 SMTP 프로토콜로 발송한다.
* **REQ-AUTH-SIGN-03 (2차 확인 및 토큰 영속성 제어):**
    * 사용자가 코드를 제출하면 `POST /api/auth/signup/email/confirm/` 세그먼트에서 정합성을 확인한다.
    * 만료 시간 이내이고 코드가 일치하는 최신 레코드를 조회하여 통과 시 테이블 내 `is_verified=True` 플래그를 영속 저장해야 한다.
* **REQ-AUTH-SIGN-04 (최종 가입 트랜잭션 및 암호화 요구사항):**
    * 최종 가입 요청(`POST /api/auth/signup/`) 처리 시, 백엔드는 입력된 `password`와 `password_confirm` 필드의 일치 여부를 대조하고 Django 내장 복잡도 검증기(`validate_password`)를 통과시켜야 한다.
    * 가입 완료 직전, 인증 테이블에서 해당 메일의 가입 인증 토큰 상태를 최종 검증한 뒤 `is_used=True`로 전환(소모 처리)하여 일회성 비즈니스 프로토콜을 만족시켜야 한다. 비밀번호는 반드시 단방향 해시 암호화(`set_password`) 처리를 거쳐 저장되어야 하며 가입 성공 시 `id`, `email`, `username`을 반환한다.

#### 2.2.2. 비밀번호 재설정(Password Reset) 보안 파이프라인 요구사항
* **REQ-AUTH-RESET-01 (역방향 계정 존재 확인 및 전용 템플릿 처리):**
    * 비밀번호 찾기 메일 요청 시 회원가입과 정반대로 `User` 테이블에 해당 메일 주소가 존재하지 않는 경우 즉시 `ValidationError("가입되지 않은 이메일입니다.")` 예외를 발생시켜야 한다.
    * 인증 코드 생성 프로토콜은 동일하게 따르되 목측 지표 필드를 `purpose='password_reset'`으로 분리 격리해야 하며, 메일 본문 템플릿 및 제목을 `"[이젠, 안쉼] 비밀번호 재설정 인증번호"` 규격으로 변경 적용해야 한다.
* **REQ-AUTH-RESET-02 (단일 엔드포인트 코드 소모 및 원자적 변경 업데이트):**
    * 비밀번호 재설정 프로세스는 프론트엔드의 화면 단계를 간소화하고 보안 위협 결합을 막기 위해, 별도의 중간 검증(Confirm) API 단계를 거치지 않고 `POST /api/auth/password-reset/` 단일 엔드포인트에서 코드 유효성 검증과 소모 처리(`is_used=True`, `is_verified=True`), 유저 패스워드 갱신 저장을 원자적(Atomic) 트랜잭션으로 일괄 처리해야 한다.

### 2.3. 지능형 하이브리드 RAG AI 추천 챗봇 시스템 (`COMP-AI`)

#### 2.3.1. 사용자 컨텍스트 주입 및 프로필 정규화 엔진 요구사항
* **REQ-AI-CTX-01 (자연어 질의 프로필 보정 결합 로직):**
    * 시스템은 비로그인 사용자 및 명시적 프로필 정보가 없는 유저의 자연어 질문도 단독 처리할 수 있어야 한다.
    * 반면 로그인된 유저가 프로필 속성(나이, 거주지역, 관심 도메인)을 가진 상태에서 질문을 제출하거나 프론트엔드가 명시적으로 프로필 구조체를 전송한 경우, 시스템은 자연어 질의 하단에 프로필 메타 문자열을 결합하여 가공해야 한다.
    * 가공 정형 포맷: `"{Original_Message}\n\n사용자 프로필 조건: 나이: {age} / 지역: {region} / 관심분야: {interests}"`
* **REQ-AI-CTX-02 (시스템 코드 매핑 사전 구동 요구사항):**
    * 프론트엔드 유입 및 세션 프로필의 유연한 처리를 위해 백엔드는 내부 시스템 매핑 테이블을 구현해야 한다.
    * 거주지역 텍스트(예: 서울시, 서울특별시, 경기도) 유입 시 공공 데이터 포털 및 시스템 인덱스 매핑 규격에 적합한 표준 행정 구역 코드(예: 서울 -> `'11000'`, 경기 -> `'41000'`)인 `region_code`로 정규화 변환해야 한다.
    * 사용자의 다중 영문 관심 키워드는 매핑 테이블(`INTEREST_LABEL_MAP`)을 경유하여 한글 표준 도메인 문자열(예: `'employment'`/`'job'` -> `'취업'`, `'housing'` -> `'주거'`)로 변환 가공되어 LangGraph 워크플로우 분석 프레임에 주입되어야 한다.

#### 2.3.2. RAG 파싱 및 스마트 조건 매칭 예외 엔진
* **REQ-AI-RAG-01 (안전성 확보를 위한 동적 인포트 패턴):**
    * 임베딩 모델 가동 라이브러리 및 AI 런타임 환경 결함으로 인해 전체 Django 웹 서버의 데몬 실행 자체가 중단되는 사태를 차단하기 위해, LangGraph 워크플로우 엔진 진입 메소드(`run_rag_workflow`) 기술 호출은 전체 파일 상단 임포트가 아닌 `run_ai_chat` 서비스 서비스 비즈니스 함수 내부 로컬 스코프 영역 내에서 동적 지연 임포트(Lazy Import) 방식으로 수행되어야 한다.
* **REQ-AI-RAG-02 (비즈니스 문자열 바운딩 제약 규칙):**
    * RAG 컴포넌트 통신 시 프론트엔드 카드 컴포넌트의 레이아웃 붕괴 및 오버플로우 방지를 위해 백엔드 파서 단에서 상수(Constant) 기반 글자수 제약 조건(Truncation)을 엄격히 적용해야 한다.
    * 최대 제약 바운더리 기준: 카드형 요약문(`CARD_SUMMARY_MAX_CHARS = 180`), 본문 지원 내용(`CARD_TEXT_MAX_CHARS = 300`), 가시적 노출 경고 리스트(`CARD_LIST_MAX_ITEMS = 3`), 하단 혜택 요약문(`BENEFIT_TEXT_MAX_CHARS = 160`). 해당 글자 수 초과 시 공백 제거 및 우측 말줄임표(`"..."`) 결합 가공을 자동 수행한다.
* **REQ-AI-RAG-03 (개인화 정합성 및 자격 매칭 역추적 알고리즘):**
    * 데이터 분석 엔진에서 가공된 조건 리스트(`matched_conditions`, `missing_conditions`, `blockers`) 유입 시 사용자의 실제 입력 프로필 존재 여부와 대조하여 필터를 역보정해야 한다.
    * 사용자가 자신의 소득 및 고용 상태 조건을 입력하지 않은 채 단순 질의를 한 경우, RAG 엔진이 해당 정책을 '소득 매칭'으로 분류했더라도 백엔드 서비스 레이어는 이를 매칭 항목에서 즉시 제거하고 '추가 확인 필요(`missing_conditions`)' 목록으로 강제 이관하여 프론트엔드에 전달해야 한다.

#### 2.3.3. 혜택 금액 자동 산출 및 금융 시각화 계산기 요구사항 (6-7 요구사항 반영)
* **REQ-AI-CALC-01 (정규식 기반 금융 텍스트 파싱 로직):**
    * 시스템은 정책의 `summary`, `support_content`, `text` 데이터를 결합한 텍스트 코퍼스 풀에서 금액 및 기간 관련 정보를 추출하는 정규식 파싱 엔진을 탑재해야 한다.
    * 금액 파싱 패턴 수식: `(?:월|매월)?\s*(?:최대\s*)?(?:\d{1,3}(?:,\d{3})+|\d+)(?:\.\d+)?\s*(?:억원|만원|천원|원)`
    * 기간 파싱 패턴 수식: `(?:최대\s*)?\d+\s*(?:개월|년|회)`
* **REQ-AI-CALC-02 (지급 주기 및 유형별 자동 산출 분기 요건):**
    * 시스템은 파싱된 금융 정보를 토대로 아래 규칙에 따라 혜택 유형(`benefit_type`)을 분류하고 총액을 동적 계산해야 한다:
        1.  **매월 현금성 지원 (`monthly_cash`):** 파싱된 금액 유효 숫자값과 지원 기간(단위: 개월, '년' 단위 유입 시 12 배수 적용 변환) 정보가 모두 존재하며 지급 주기가 '월' 단위임이 검증된 경우, `benefit_estimate_available = True` 플래그를 수립하고 `금액 * 개월수` 연산을 처리하여 `max_total_benefit_text` 필드에 `"최대 X만원"` 형태로 바인딩한다. 수식 연산의 투명성을 위해 `benefit_calculation_text` 필드에 실제 대입 수식문(예: `"월 20만원 × 12개월 = 최대 240만원"`)을 생성하여 반환해야 한다.
        2.  **임차보증금 무이자 지원 (`interest_free_deposit`):** 텍스트 코퍼스 내 "무이자" 또는 "보증금" 관련 도메인이 탐지되는 경우 산출 불가능 유형으로 식별하여 계산 가능 플래그를 `False`로 고정하고, `max_total_benefit_text` 필드에 `"무이자 지원"` 텍스트를 할당하며, 안내 메시지 영역에 개인별 보증금 규모에 따른 계산 예외 고지문을 자동 주입해야 한다.
        3.  **기타/정보 불충분 유형 (`cash` / `unknown`):** 고정 단위 기간 매칭이 실패하거나 지원 단위가 명확하지 않은 경우, 계산 가용 상태를 `False`로 선언하고 프론트엔드 UI 영역에 `"원문 확인 필요"` 리터럴과 수동 확인 권장 고지 배너 정보를 전달해야 한다.

```
[텍스트 코퍼스 풀 정밀 스캔]
         │
         ▼
 (유형 판별 분기) ──> [1. 월 단위 현금 수당] ──> 수식 연산 작동 (금액 × 개월) ──> "최대 240만원" 반환
         │
         ├───> [2. 보증금/무이자 대출] ──> 계산 불가 처리 ───────────────> "무이자 지원" 반환
         │
         └───> [3. 정보 불명확/단발성] ──> 계산 불가 처리 ───────────────> "원문 확인 필요" 반환
```

#### 2.3.4. 신청 가능성 판별 지표 및 상태 뱃지 동기화 요구사항
* **REQ-AI-STAT-01 (3단계 매칭 자격 레이블 및 CSS 타입 제어 스펙):**
    * 백엔드는 각 정책 추천 항목별로 자격 충족 신뢰도 점수를 계산하여 프론트엔드 UI에 바인딩할 3가지 명확한 시각 상태 지표 파라미터를 계산 출력해야 한다.
    * 지표 산출 매트릭스 세부 조건:
        1.  **`high` 상태 (신청 가능성 높음):** 정책 마감 상태가 마감이 아니고, 자격 불충분 걸림돌(`blockers`)이 없으며, RAG 판단 결과 통과 신뢰도가 높은 경우 레이블 필드에 `"가능성 높음"`, 뱃지 텍스트에 `"가능성 높음"`, 프론트엔드 Tailwind/CSS 스타일 바인딩용 뱃지 타입을 `eligibility_badge_type = 'success'`(초록색)로 지정 출력한다.
        2.  **`needs_check` 상태 (추가 확인 필요):** 자격 걸림돌이 감지되지 않았으나 소득, 주거 등 사용자 입력 항목 부재로 추가 역추적 대조 확인이 필요한 경우 레이블 필드에 `"추가 확인 필요"`, 뱃지 텍스트에 `"확인 필요"`, CSS 뱃지 타입을 `eligibility_badge_type = 'warning'`(노랑/주황색)으로 선언 출력한다.
        3.  **`low` 상태 (신청 어려움):** 원천 데이터 마감 플래그(`is_expired = True` 또는 `deadline_status = 'expired'`)가 감지되거나 자격 불충분 항목(`blockers`)이 실질적으로 판명된 경우 레이블 필드에 `"가능성 낮음"`, 뱃지 텍스트에 `"신청 어려움"`, CSS 뱃지 타입을 `eligibility_badge_type = 'danger'`(빨강색)로 강제 지정 출력한다.
    * 프론트엔드 가시성 보장을 위해 각 정책 카드 내 자격 확인 정보 체크 리스트 항목(`eligibility_check_items`)을 구조체 배열 구조(`[{label: '나이', status: 'matched', display_status: '충족 가능'}]`)로 정형화하여 상위 5건까지 제한 슬라이싱하여 UI에 바인딩 피딩해야 한다.

#### 2.3.5. 글로벌 폴백 및 장애 격리(Fault Tolerance) 행위 요구사항
* **REQ-AI-ERR-01 (시스템 에러 팩토리 구축 표준):**
    * 챗봇 질의 처리 중 발생하는 기술적 장애 유형에 따라 내부 로깅 시스템에 추적 로그를 남기고, 프론트엔드가 크래시(Crash) 없이 정상 동작할 수 있도록 표준화된 에러 페이로드 규격의 데이터 구조체를 응답해야 한다.
    * 반드시 분기 처리해야 할 예외 코드 매트릭스:
        * `EMPTY_MESSAGE`: 사용자의 텍스트 입력을 확인하지 못한 경우
        * `WORKFLOW_PARSE_ERROR`: RAG 그래프 결과 딕셔너리 파싱 구조 오염 발생 시
        * `AI_CONNECTION_ERROR`: 외부 LLM 서버 및 통신 소켓 다운 발생 시
        * `LLM_RATE_LIMIT`: OpenAI API 429(Too Many Requests) 핸들링 유입 시
        * `LLM_TIMEOUT`: 지정된 타임아웃 임계 임계치 시간 초과 시
        * `LLM_AUTH_ERROR`: 서버 API Key 서명 실패 혹은 권한 박탈 시
        * `VECTOR_DB_ERROR`: Chroma DB 디스크 I/O 장애 및 컬렉션 분실 시
    * 장애 발생 시 백엔드는 전면 시스템 에러를 표출하는 대신 데이터 영역 내 `fallback_used = True` 값을 설정하고 `FALLBACK_ANSWER`(`"죄송합니다. 현재 정책 검색 서비스에 일시적인 문제가 발생했습니다..."`) 문장 문자열을 `answer` 필드에 밀어 넣어 반환함으로써 플랫폼의 가용성(Availability)을 보장해야 한다.

### 2.4. 통합 도메인 인프라 및 사용자 활동 트래킹 시스템 (`COMP-DATA`)

#### 2.4.1. 다중 출처 공공 데이터 통합 아키텍처 요구사항
* **REQ-DATA-INT-01 (3대 공공 데이터 포털 수집 스키마 일원화 단일 테이블 설계):**
    * 온통청년(청년정책), K-Startup(창업공고), 고용24(훈련과정) 등 수집 출처에 따라 상이한 원본 데이터 모델 구조를 영속성 레이어 내 단일 테이블 스키마 구조인 `Policy` (물리 테이블명: `policies`) 테이블로 통합 일원화 정형 적재해야 한다.
    * 중복 적재 감지 및 수집 주기 변경 추적 관리를 위해 원본 출처 식별 ID 키값인 `original_id` 필드에 데이터베이스 인덱스(`db_index=True`)를 설정해야 한다.
    * 출처 카테고리 필드 `source_category`는 반드시 `['policy', 'startup_notice', 'training']` 도메인 범위의 Choices 제약을 적용하여 무결성을 유지해야 한다.
* **REQ-DATA-INT-02 (원본 데이터 보존 및 메타데이터 필드 요건):**
    * 다양한 원본 필드의 유실을 차단하기 위해 스키마 구조 내에 정형화되지 않은 나머지 데이터 속성 전체를 압축 보존할 수 있는 확장용 JSON 데이터 필드(`raw_data = models.JSONField()`)를 배치 설계해야 한다.
    * 정승 매니저 모듈이 관장하는 데이터 배치 및 실시간 동기화 주기 엔진은 마감 상태 지표 속성 필드인 `deadline_status` 값을 실시간 대조하여 `['upcoming', 'ongoing', 'closing_soon', 'closed', 'unknown']` 중 하나로 업데이트 싱크를 유지해야 한다.

#### 2.4.2. 사용자 플랫폼 활동 로그 및 개인화 빅데이터 수집 요구사항
* **REQ-DATA-LOG-01 (스크랩 및 이력 추적 관계 관계성 스키마 요건):**
    * 사용자별 맞춤형 추천 플랫폼 활동 조회를 지원하기 위해 백엔드 인프라는 총 3개 영역의 유저 이력 트래킹 관계 테이블을 실시간 로깅 작동시켜야 한다:
        1.  **북마크 스크랩 관리 테이블 (`Scrap` / `scraps`):** 로그인 유저와 `Policy` 테이블 간의 다대다(M:N) 관계 교차 스키마로 작동한다. 동일 유저가 동일 정책을 중복 스크랩하는 무결성 오염을 차단하기 위해 데이터베이스 단에 복합 유니크 제약 조건(`unique_together = ('user', 'policy')`)을 필수 강제 적용하고, 역참조 역인덱스 정렬 관계는 최근 스크랩 순서(`-created_at`)로 선언 처리한다.
        2.  **최근 본 공고 이력 테이블 (`ViewedPolicy` / `viewed_policies`):** 사용자가 상세 화면을 조회할 때마다 진입 타임스탬프 기록을 `viewed_at = models.DateTimeField(auto_now=True)` 필드 속성을 통해 실시간 업데이트하며, 유저별 복합 유니크 제약 조건을 부여하여 행 비대화를 방지한다.
        3.  **검색 히스토리 아카이브 테이블 (`SearchHistory` / `search_history`):** 유저가 메인 검색 바에 입력한 텍스트 문자열 데이터를 저장하며, 검색 시점의 역순 정렬 스펙을 유지한다.

#### 2.4.3. 개인정보 격리형 실시간 인기 검색어 집계 시스템 요구사항
* **REQ-DATA-STAT-01 (개인정보 격리형 글로벌 통계 익명 테이블 설계):**
    * 플랫폼 내 실시간 트렌드 지표인 인기 검색어 순위를 도출하기 위해, 시스템은 사용자 식별 세션 키 정보(`user_id`, `email`, `ip`)를 완전 배제하고 오직 입력 키워드 문자열 정보만을 수집하는 격리형 독립 통계 테이블 `PopularSearchKeyword` (물리 테이블명: `popular_search_keywords`)를 별도 설계 가동해야 한다.
    * 성능 보장 및 랭킹 정렬 쿼리 부하 최소화를 위해 중복을 방지하는 키워드 필드(`keyword`)를 고유키(`unique=True`) 및 인덱스 처리해야 하며, 정렬 속도 극대화를 위해 집계 횟수와 키워드 명을 묶은 복합 고속 인덱스(`models.Index(fields=['-count', 'keyword'])`)를 물리 저장소 레벨에 선언 가동해야 한다.

## 4. 데이터베이스 엔티티 상세 스키마 설계 (Database Schema)

시스템의 영속성 계층을 구성하는 핵심 테이블의 릴레이션 및 데이터 무결성 제약 조건 명세입니다.

### 4.1. `EmailVerificationCode` 테이블 (물리 테이블명: `email_verification_codes` 권장)
인증 세션 제어 및 토큰 보안 처리를 위한 저장소 아키텍처 사양입니다.

| 필드명 (Field) | 데이터 타입 (Type) | 제약 조건 (Constraints) | 설명 (Description) |
| :--- | :--- | :--- | :--- |
| `id` | `BigAutoField` | `PRIMARY KEY`, `AUTO_INCREMENT` | 고유 인덱스 식별 키 값 |
| `email` | `EmailField(max_length=254)` | `NOT NULL`, `db_index=True` | 인증 대상 사용자 이메일 주소 |
| `code` | `CharField(max_length=6)` | `NOT NULL`, `MinLengthValidator(6)` | 스트링 패딩 처리된 6자리 숫자형 난수 코드 |
| `purpose` | `CharField(max_length=20)` | `NOT NULL`, `Choices 분기` | `signup`(회원가입), `password_reset`(비밀번호 재설정) |
| `is_verified` | `BooleanField` | `NOT NULL`, `Default=False` | 2차 Confirm API 인증 통과 완료 여부 |
| `is_used` | `BooleanField` | `NOT NULL`, `Default=False` | 트랜잭션 최종 소모 처리 및 폐기 상태 플래그 |
| `expires_at` | `DateTimeField` | `NOT NULL` | 인증 코드의 비즈니스 제한 만료 시점 타임스탬프 |
| `created_at` | `DateTimeField` | `NOT NULL`, `auto_now_add=True` | 최초 코드 인서트 발급 타임스탬프 |
| `verified_at` | `DateTimeField` | `NULL=True`, `Blank=True` | 실질적 인증 승인 처리 처리 시점 로그 |

* **복합 인덱스 설계 요건 (Composite Index):** 이메일 기반 역추적 쿼리 속도 보장을 위해 최우선 인덱스 제약을 선언한다: `[email, purpose, -created_at]` 순으로 결합된 복합 조회 인덱스(`users_email_verify_lookup`)를 구성해야 한다.

### 4.2. `Policy` 테이블 (물리 테이블명: `policies`)
공공 인프라 3대 기관 포털 수집 데이터를 가공 통합한 코어 마스터 엔티티 스키마입니다.

| 필드명 (Field) | 데이터 타입 (Type) | 제약 조건 (Constraints) | 설명 (Description) |
| :--- | :--- | :--- | :--- |
| `item_id` | `CharField(max_length=100)` | `PRIMARY KEY` | 시스템 통합 관리 고유 키 (예: policy_2026...) |
| `original_id` | `CharField(max_length=100)` | `NOT NULL`, `db_index=True` | 공공 출처 원본 ID (온통청년 / K-Startup / 고용24) |
| `source_category` | `CharField(max_length=30)` | `NOT NULL`, `db_index=True` | `policy`(정책), `startup_notice`(창업), `training`(훈련) |
| `source_name` | `CharField(max_length=100)` | `Blank=True`, `Default=""` | 상세 운영 플랫폼 한글명 (예: 온통청년) |
| `title` | `CharField(max_length=300)` | `NOT NULL` | 지원 정책 및 공고 한글 명칭 |
| `domain` | `CharField(max_length=200)` | `Blank=True` | 지원 세부 분야 트리 구조 문자열 |
| `policy_summary` | `TextField` | `Blank=True` | 정책 요약 핵심 본문 데이터 |
| `participation_target`| `TextField` | `Blank=True` | 지원 자격 및 참여 대상 상세 요건 설명문 |
| `benefit_text` | `TextField` | `Blank=True` | 실제 지원금 규모 및 금융 혜택 원문 텍스트 |
| `region_codes` | `JSONField` | `Default=list` | 지원 대상 행정 구역 코드 리스트 구조 배열 |
| `income_condition` | `TextField` | `Blank=True` | 소득 자격 요건 제한 기술문 |
| `age_min` | `IntegerField` | `NULL=True` | 지원 대상 하한 연령 기준 제한값 (Null 일 시 제한 없음) |
| `age_max` | `IntegerField` | `NULL=True` | 지원 대상 상한 연령 기준 제한값 (Null 일 시 제한 없음) |
| `application_period_text`| `TextField`| `Blank=True` | 신청 기간 원문 기술 텍스트 |
| `application_start_date`| `DateField` | `NULL=True` | 정형화 변환된 접수 시작일 |
| `application_end_date` | `DateField` | `NULL=True` | 정형화 변환된 접수 최종 마감일 |
| `application_url` | `URLField(max_length=500)`| `Blank=True` | 다이렉트 프론트 아웃링크 접수 URL 웹 주소 |
| `source_url` | `URLField(max_length=500)`| `Blank=True` | 공공 포털 원문 상세 화면 상세 이동 주소 |
| `raw_data` | `JSONField` | `Default=dict` | 원본 필드 스키마 유실 대비용 전체 데이터 백업 보존 필드 |
| `deadline_status` | `CharField(max_length=20)` | `NOT NULL`, `db_index=True` | 정승 담당 갱신 지표 (`ongoing`, `closed`, `unknown` 등) |
| `created_at` | `DateTimeField` | `auto_now_add=True` | 시스템 최초 적재 적재 일자 |
| `updated_at` | `DateTimeField` | `auto_now=True` | 배치 동기화 최종 변경 타임스탬프 |
