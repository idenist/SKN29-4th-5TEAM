* 폼 검증 에러 처리 및 API 레이턴시 제어를 통한 안정적 서비스 구동을 확인했습니다.

### 🤖 AI/RAG 영역 상세 결과 (윤승혁 제공, 2026-07-08)

**① AI Chat API 정상 응답 (`POST /api/ai/chat/`)**
질문 "2026년에 신청 가능한 창업지원 공고 추천해줘"에 대해 `success=true`, `recommendations=5개`, `sources=5개`, `warnings=[]`, `error=null`로 정상 응답했습니다. 내부 trace 기준 route=창업, source_category=startup_notice, expired_count=0, open_count=5로 마감 공고 없이 신청 가능한 공고만 반환되었으며, 검색 결과가 충분하다고 판단되어(`internal_search_sufficient=true`) 외부 검색(Tavily) fallback 없이 내부 RAG만으로 답변이 완료되었습니다.

**② 창업공고 현재성 검색 개선**
`/api/policies/` 목록 API에서는 신청 가능한 창업공고가 조회되는데 `/api/ai/chat/`에서는 0건으로 반환되는 불일치가 있었습니다. 원인은 RAG 검색 필터가 `domain=창업`을 사용했으나 Chroma metadata의 실제 값은 `domain=startup`이었던 점, 그리고 semantic search 상위 후보에 진행 중인 공고가 포함되지 않을 경우 현재성 필터 적용 후 결과가 0건이 되는 구조였습니다. `source_category=startup_notice` 기준 검색으로 전환하고, 현재성 질문에서 `fetch_k`를 확대했으며, semantic search 결과 부족 시 `application_end_date >= today` 기준 metadata fallback을 추가하여 개선했습니다. 개선 후 `retrieve_policies()` 직접 호출과 `/api/ai/chat/` 호출 모두 `count/recommendations=5`, `expired_count=0`, `open_count=5`로 일치하는 결과를 확인했습니다.

**③ 예상 지원금 계산 (`tests/test_benefit_calculation.py`)**

## 10. 최종 한계 및 개선 방향
3차 프로젝트에서 도출된 기술적 단점과 4차 통합 테스트 과정을 거치며 발견된 인프라 및 UI 레벨의 한계점, 그리고 이번 업데이트에서 새로 확인된 백엔드 운영 리스크를 정리합니다.

* **[한계점 1] ReAct 에이전트 다중 호출로 인한 응답 지연(Latency):** `Eligibility Checker`와 `Result Sufficiency Checker`가 LangGraph 워크플로우 상에서 반복 구동됨에 따라 챗봇 최종 답변까지 평균 5~8초의 지연 발생.
  * **➡️ 개선 방향:** 백엔드 API 단에서 SSE(Server-Sent Events) 기반 스트리밍 응답을 5차 고도화 규격으로 반영, 프론트엔드에 실시간 추론 단계 인디케이터 UI 신설 계획.
* **[한계점 2] 수치형 제약조건(Hard Constraints) 필터링 환각 가능성:** RAG 벡터 검색이 "중위소득 120% 이하" 같은 엄격한 수치 조건을 100% 걸러내지 못해 발생하는 정보 노이즈 잔존.
  * **➡️ 개선 방향:** Django ORM 레벨의 1차 물리 필터링(Hard Filtering) 구조를 정교화하여 LLM 순수 추론 의존도를 낮출 계획. *(참고: LLM/API 실패·타임아웃 등 예외 상황에 대한 fallback 처리 자체는 TC-AIEXC-01~05 검증을 통해 안정성이 확인되었으며, 본 한계점은 예외 처리가 아닌 수치 조건 필터링 정확도에 국한된 사항입니다.)*
* 🆕 **[한계점 3] EC2 배포 파이프라인의 수동화에 따른 지연 리스크:** CI/CD가 아직 착수되지 않아, 로컬 검증 완료 항목이 `main`에 push된 이후에도 EC2 반영까지 수동 배포 타이밍에 의존. 현재 팀 선호는 `workflow_dispatch` 수동 트리거이며, 평가 일정이 촉박할 경우 단순 `deploy.sh`로 대체 가능. CI/CD가 평가 필수 항목인지는 미확인 상태.
  * **➡️ 개선 방향:** `ec2_deploy_final.sh` 기반 단일 배포 패스를 우선 실행하고, 이후 필요 시 `workflow_dispatch` 형태의 수동 CI/CD를 추가.
* 🆕 **[한계점 4] 추천(Recommendations) 앱 구현 상태 미확인:** `apps/recommendations` (윤승혁 담당)의 `urls.py`가 아직 TODO 스켈레톤 상태로, 구현 완료 여부가 백엔드 리드 기준으로 확인되지 않음.
  * **➡️ 개선 방향:** 윤승혁과 구현 상태를 조속히 확인하고, 테스트 케이스(TC-AI 시리즈)에 반영.
* 🆕 **[한계점 5] 마감 알림 스케줄러 미구성:** `generate_deadline_notifications` management command 자체는 정상 동작하나, 이를 주기적으로 실행할 cron 또는 Celery 스케줄러가 아직 설정되지 않아 현재는 수동 실행에 의존.
  * **➡️ 개선 방향:** EC2 crontab 또는 Celery beat 스케줄러를 배포 패스에 포함시켜 자동화.
