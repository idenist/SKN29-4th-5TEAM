# sLLM LoRA/QLoRA 파인튜닝 실험 보고서 보강본

## 1. 실험 목적

본 실험은 청년 정책 통합 탐색 에이전트 프로젝트에서 sLLM 기반 파인튜닝 가능성을 확인하기 위해 수행하였다. 운영 서비스의 메인 구조는 OpenAI 기반 RAG 파이프라인을 유지하되, 평가 기준의 `sLLM 파인튜닝 및 최적화 코드 역량` 항목에 대응하기 위해 LoRA/QLoRA 기반 PEFT 파인튜닝 코드와 샘플 학습·추론 결과를 구성하였다.

본 실험의 핵심 목적은 다음과 같다.

```text
1. 정책/창업공고/교육훈련 데이터를 instruction tuning 형식으로 변환할 수 있는지 확인
2. LoRA 또는 QLoRA 방식으로 adapter 학습 및 저장이 가능한지 확인
3. 학습된 adapter를 base model과 결합하여 한국어 정책형 응답을 생성할 수 있는지 확인
4. sLLM 단독 응답 방식의 한계와 운영 RAG 파이프라인과의 역할 차이를 정리
5. 향후 RAG context 기반 sLLM 답변 생성기 또는 조건 추출기 확장 가능성 검토
```

## 2. 실험 산출물 및 구현 파일

### 2.1 주요 스크립트

| 파일 | 역할 |
|---|---|
| `sllm/scripts/prepare_sllm_dataset.py` | 전처리된 정책/창업공고/교육훈련 데이터를 instruction tuning용 JSONL로 변환 |
| `sllm/scripts/train_lora.py` | YAML config를 기반으로 LoRA 또는 QLoRA adapter 학습 수행 |
| `sllm/scripts/evaluate_sllm.py` | 학습 adapter의 응답을 생성하고 ROUGE-L, 키워드 적중률, 출처/확인 안내 포함률 등을 산출하는 정량 평가 스크립트 |
| `sllm/scripts/run_lm_eval.py` | `lm-evaluation-harness` 실행 래퍼. 표준 벤치마크 또는 별도 task yaml 준비 시 사용 |

### 2.2 설정 및 결과 파일

| 파일 | 내용 |
|---|---|
| `sllm/configs/lora_config.yaml` | 일반 LoRA 학습 설정 |
| `sllm/configs/qlora_config.yaml` | 4bit quantization 기반 QLoRA 학습 설정 |
| `sllm/data/policy_qa_train_sample.jsonl` | 학습용 instruction 데이터셋 |
| `sllm/data/policy_qa_eval_sample.jsonl` | 평가용 instruction 데이터셋 |
| `sllm/outputs/lora_policy_qa` | LoRA adapter 저장 경로 |
| `sllm/outputs/qlora_policy_qa` | QLoRA adapter 저장 경로 |
| `sllm_prompt_test_results.jsonl` | 5개 대표 프롬프트 생성 테스트 결과 |

> 주의: 본 보강본 작성 시점에는 `sllm/evaluation/sllm_eval_results.json` 파일이 없으므로, ROUGE-L F1, keyword hit rate 등 정량 평가 결과는 최종 수치로 보고하지 않는다. 대신 `evaluate_sllm.py`는 향후 정량 평가를 위한 준비된 스크립트로 설명하고, 실제 결과 해석은 `sllm_prompt_test_results.jsonl` 기반의 5개 프롬프트 생성 테스트를 중심으로 수행한다.

## 3. 데이터셋 생성 방식

`sllm/scripts/prepare_sllm_dataset.py`는 다음 입력 후보를 처리할 수 있도록 작성되었다.

```text
data/processed/opportunities.json
data/processed/policies.json
data/processed/chunks.jsonl
data/processed/opportunity_chunks.jsonl
```

출력 파일은 다음과 같다.

```text
sllm/data/policy_qa_train_sample.jsonl
sllm/data/policy_qa_eval_sample.jsonl
```

### 3.1 instruction 데이터 구조

생성되는 각 row는 다음 구조를 가진다.

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "질문 + 참고 정책 정보"},
    {"role": "assistant", "content": "정답 답변"}
  ],
  "instruction": "사용자 질문",
  "input": "제목/유형/분야/본문/출처",
  "output": "정답 답변",
  "metadata": {
    "source_category": "policy",
    "domain": "주거",
    "title": "...",
    "source_url": "..."
  }
}
```

### 3.2 source_category별 질문 템플릿

데이터 유형별로 다른 질문 템플릿을 적용한다.

| source_category | 질문 템플릿 예시 |
|---|---|
| `policy` | `{title} 정책에 대해 알려줘`, `{title} 신청 조건과 지원 내용을 알려줘` |
| `startup_notice` | `{title} 창업지원 공고 알려줘`, `예비창업자가 볼 만한 {title} 공고 내용을 알려줘` |
| `training` | `{title} 교육훈련 과정 알려줘`, `국민내일배움카드로 들을 수 있는 {title} 과정 설명해줘` |

### 3.3 시스템 메시지 원칙

데이터셋 생성 시 system message에는 다음 원칙이 포함된다.

```text
너는 청년 정책 추천 RAG 서비스의 답변 생성기다.
제공된 정책 정보에 근거해서만 답변하고, 정보가 없으면 없다고 말한다.
신청 기간, 제출 서류, 자격 조건은 확정적으로 단정하지 말고 추가 확인 필요 여부를 함께 안내한다.
```

이는 운영 RAG 파이프라인의 출처 기반 답변 원칙을 sLLM 학습 데이터에도 반영하기 위한 장치이다.

### 3.4 데이터셋 생성 명령어 예시

```bash
python sllm/scripts/prepare_sllm_dataset.py   --input data/processed/opportunities.json   --output-dir sllm/data   --max-samples 1000   --eval-ratio 0.1   --seed 42
```

## 4. LoRA/QLoRA 학습 설정

본 프로젝트의 `train_lora.py`는 YAML 설정 파일을 읽어 LoRA 또는 QLoRA 학습을 수행한다. `use_qlora=false`이면 일반 LoRA 학습을 수행하고, `use_qlora=true`이면 4bit quantization 기반 QLoRA 경로를 사용한다.

### 4.1 공통 학습 설정

| 항목 | 값 |
|---|---|
| Base Model | `Qwen/Qwen2.5-1.5B-Instruct` |
| train_file | `sllm/data/policy_qa_train_sample.jsonl` |
| eval_file | `sllm/data/policy_qa_eval_sample.jsonl` |
| seed | `42` |
| max_seq_length | `1024` |
| torch_dtype | `bfloat16` |
| device_map | `auto` |
| num_train_epochs | `1` |
| per_device_train_batch_size | `1` |
| per_device_eval_batch_size | `1` |
| gradient_accumulation_steps | `8` |
| learning_rate | `0.0002` |
| weight_decay | `0.0` |
| warmup_ratio | `0.03` |
| lr_scheduler_type | `cosine` |
| max_grad_norm | `1.0` |
| gradient_checkpointing | `true` |
| bf16 | `true` |
| fp16 | `false` |
| save_steps | `100` |
| eval_steps | `100` |
| save_total_limit | `2` |
| report_to | `none` |

### 4.2 LoRA 설정

| 항목 | 값 |
|---|---|
| output_dir | `sllm/outputs/lora_policy_qa` |
| use_qlora | `false` |
| lora_r | `16` |
| lora_alpha | `32` |
| lora_dropout | `0.05` |
| target_modules | `q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj` |

LoRA는 전체 모델 파라미터를 모두 업데이트하지 않고, 지정된 attention/MLP projection module에 adapter를 삽입하여 학습한다. 이 방식은 전체 파인튜닝보다 GPU 메모리 요구량이 낮고, 학습 결과를 adapter 형태로 분리 저장할 수 있다는 장점이 있다.

### 4.3 QLoRA 설정

| 항목 | 값 |
|---|---|
| output_dir | `sllm/outputs/qlora_policy_qa` |
| use_qlora | `true` |
| load_in_4bit | `true` |
| bnb_4bit_quant_type | `nf4` |
| bnb_4bit_compute_dtype | `bfloat16` |
| bnb_4bit_use_double_quant | `true` |
| lora_r | `16` |
| lora_alpha | `32` |
| lora_dropout | `0.05` |

`train_lora.py`는 `use_qlora=true`일 때 `BitsAndBytesConfig`를 사용하여 4bit quantization을 적용하고, `prepare_model_for_kbit_training()`으로 k-bit 학습 준비를 수행한다. 따라서 GPU 메모리가 부족한 환경에서는 QLoRA 설정을 사용해 VRAM 사용량을 줄일 수 있다.

### 4.4 학습 실행 명령어

LoRA 학습:

```bash
python sllm/scripts/train_lora.py --config sllm/configs/lora_config.yaml
```

QLoRA 학습:

```bash
python sllm/scripts/train_lora.py --config sllm/configs/qlora_config.yaml
```

## 5. 평가 및 테스트 방식

### 5.1 프롬프트 생성 테스트

이번 산출물에서 실제 결과가 존재하는 테스트는 `sllm_prompt_test_results.jsonl` 기반 5개 프롬프트 생성 테스트이다.

| 항목 | 값 |
|---|---|
| 테스트 프롬프트 수 | `5` |
| 정상 생성 수 | `5` |
| 실패 수 | `0` |
| 평균 생성 길이 | `513.2`자 |
| Base Model | `Qwen/Qwen2.5-1.5B-Instruct` |
| Adapter Path | `sllm/outputs/lora_policy_qa` |
| 최초 테스트 시각 | `2026-06-15T09:20:01` |

### 5.2 테스트 케이스별 결과

| Case | Prompt | 생성 길이 | 실행 결과 | 주요 관찰 |
|---|---|---:|---|---|

| TC002 | 국민내일배움카드로 들을 수 있는 AI 데이터 분석 훈련 과정을 추천해줘. | 878 | 정상 생성 | 교육훈련 과정 형식은 따랐으나 동일 표현 반복 생성 발생 |
| TC003 | 2026년에 신청 가능한 청년 창업지원 공고를 알려줘. | 325 | 정상 생성 | 2026년 질문에 2025년 신청기간을 포함하여 최신성 판단 미흡 |
| TC004 | 취업준비생이 받을 수 있는 면접수당 정책이 있나요? | 311 | 정상 생성 | 2019년 정책을 추천하여 오래된 정책 배제 실패 |
| TC005 | 청년도약계좌 같은 청년 금융 정책을 설명해줘. | 745 | 정상 생성 | 금융 정책 설명 형식은 따랐으나 지원 내용 반복 발생 |

### 5.3 정량 평가 스크립트

`sllm/scripts/evaluate_sllm.py`는 향후 정량 평가를 위해 준비한 스크립트이다. 해당 스크립트는 파인튜닝 adapter를 로드하여 평가 JSONL에 대한 응답을 생성하고, 다음 지표를 계산한다.

| 지표 | 설명 |
|---|---|
| `rouge_l_f1` | reference 답변과 생성 답변의 ROUGE-L F1 |
| `keyword_hit` | 정책명, 도메인, source_category 등 핵심 키워드 포함 여부 |
| `mentions_source_or_check` | `출처` 또는 `확인` 안내 포함 여부 |
| `generated_length` | 생성 답변 길이 |
| `avg_rouge_l_f1` | 전체 평균 ROUGE-L F1 |
| `keyword_hit_rate` | 키워드 적중률 |
| `source_or_check_rate` | 출처/확인 안내 포함률 |
| `avg_generated_length` | 평균 생성 길이 |

다만 본 제출 산출물에는 `sllm/evaluation/sllm_eval_results.json`이 생성되어 있지 않으므로, 위 지표의 최종 수치는 보고하지 않는다. 본 보고서에서는 5개 프롬프트 생성 테스트 결과를 기준으로 정성 평가를 수행한다.

정량 평가 실행 예시는 다음과 같다.

```bash
python sllm/scripts/evaluate_sllm.py   --base-model Qwen/Qwen2.5-1.5B-Instruct   --model sllm/outputs/lora_policy_qa   --eval-file sllm/data/policy_qa_eval_sample.jsonl   --output sllm/evaluation/sllm_eval_results.json
```

### 5.4 LM-Eval 래퍼

`sllm/scripts/run_lm_eval.py`는 `lm-evaluation-harness` 실행을 위한 래퍼이다. 표준 벤치마크 또는 별도 task yaml이 준비된 경우 사용할 수 있다.

```bash
python sllm/scripts/run_lm_eval.py   --model hf   --model-args pretrained=sllm/outputs/lora_policy_qa   --tasks kobest_boolq,kobest_hellaswag   --output-path sllm/evaluation/lm_eval_results.json
```

단, 정책 추천형 QA 품질은 일반 언어 능력 벤치마크보다 출처 정확성, 최신성, 마감 여부, 원문 근거성이 더 중요하다. 따라서 본 프로젝트에서는 `evaluate_sllm.py`와 RAG LLM-as-a-Judge 평가를 함께 해석하는 방향이 더 적절하다.

## 6. 주요 관찰 결과

### 6.1 정상 동작한 부분

```text
1. LoRA adapter 로드 및 추론 흐름을 구성하였다.
2. 5개 테스트 프롬프트 모두 error=null로 응답 생성에 성공하였다.
3. 한국어 정책 안내형 문장을 생성할 수 있었다.
4. 청년 정책, 창업지원 공고, 교육훈련 과정의 답변 형식을 일부 학습하였다.
5. 정책명, 분야, 신청기간, URL, 출처 등 구조화된 출력 패턴을 일부 생성하였다.
6. LoRA/QLoRA 학습 스크립트가 PEFT adapter 저장 구조를 갖추고 있다.
```

### 6.2 품질상 한계

```text
1. 월세 지원 질문에 K-Startup URL을 생성하는 등 도메인과 출처가 불일치하였다.
2. 2026년에 신청 가능한 창업공고 질문에 2025년 신청기간을 포함하여 최신성 판단에 실패하였다.
3. 취업준비생 면접수당 질문에 2019년 정책을 추천하여 오래된 정책 배제에 실패하였다.
4. AI 데이터 분석 훈련 과정과 청년도약계좌 설명에서 반복 생성이 발생하였다.
5. RAG 검색 context 없이 학습 패턴만으로 URL과 세부 내용을 생성하여 hallucination 위험이 확인되었다.
6. sLLM 단독 생성 방식만으로는 마감 여부, 출처 정확성, 원문 근거성을 안정적으로 보장하기 어렵다.
```

## 7. 운영 RAG 파이프라인과의 비교

| 구분 | OpenAI 기반 RAG 파이프라인 | sLLM LoRA/QLoRA 실험 모델 |
|---|---|---|
| 목적 | 실제 서비스 응답 생성 | 파인튜닝 가능성 및 PEFT 코드 역량 검증 |
| 입력 | 사용자 질문 + Vector DB 검색 context | 사용자 질문 또는 학습된 패턴 |
| 최신성 | Vector DB, deadline_status, 외부 출처 fallback으로 보완 가능 | 학습 데이터 시점에 의존 |
| 출처 정확성 | 검색 결과의 source_url/application_url 기반 | 잘못된 URL 생성 가능 |
| 마감 공고 배제 | `deadline_status`, `is_expired` 기반 처리 가능 | 오래된 정책 생성 가능 |
| 반복 제어 | 프롬프트와 후처리로 제어 가능 | decoding 설정에 따라 반복 발생 가능 |
| 환각 방지 | 검색 context 기반 답변 제한 가능 | 학습 패턴 기반 생성으로 환각 위험 존재 |
| 적합한 활용 | 운영 서비스 메인 응답 | 조건 추출기, 요약기, 답변 초안 생성기 등 보조 모듈 |

## 8. 평가 기준 대응

본 실험은 평가 기준 중 `sLLM 파인튜닝 및 최적화 코드 역량` 항목에 대응한다.

| 평가 요소 | 대응 내용 |
|---|---|
| LoRA/QLoRA 기법 활용 | `train_lora.py`에서 LoRA와 QLoRA 설정 모두 지원 |
| PEFT 파인튜닝 코드 | `peft.LoraConfig`, `get_peft_model`, `PeftModel` 기반 adapter 학습/로드 구조 구성 |
| 메모리 효율성 고려 | QLoRA 설정에서 4bit quantization, NF4, double quantization 지원 |
| 학습 안정성 | gradient accumulation, gradient checkpointing, max_grad_norm 설정 |
| 데이터셋 구성 | `prepare_sllm_dataset.py`로 정책/창업/교육훈련 데이터를 instruction format으로 변환 |
| 평가 코드 | `evaluate_sllm.py`, `run_lm_eval.py`로 정량 평가 및 표준 벤치마크 실행 기반 마련 |
| 한계 분석 | sLLM 단독 응답의 최신성, 출처 정확성, 반복 생성, 환각 위험을 분석 |

## 9. 개선 방향

이번 실험 결과를 바탕으로 다음 개선 방향을 도출하였다.

```text
1. sLLM 단독 QA 방식이 아니라 RAG 검색 context를 input에 포함한 supervised fine-tuning 데이터로 재구성한다.
2. 학습 데이터에서 마감 공고, 오래된 정책, 부정확한 URL을 제거하거나 별도 라벨링한다.
3. 출력 형식을 정책명, 지원내용, 신청기간, 출처, 확인 필요 사항으로 고정한다.
4. repetition_penalty, no_repeat_ngram_size, max_new_tokens 조정을 통해 반복 생성을 완화한다.
5. evaluate_sllm.py를 실행하여 ROUGE-L, keyword_hit_rate, source_or_check_rate를 실제 산출한다.
6. 운영 서비스에는 바로 적용하지 않고 조건 추출, 요약, 답변 초안 생성 등 제한된 보조 기능부터 적용한다.
7. 장기적으로는 검색 context를 받은 sLLM이 원문 기반 답변만 생성하도록 RAG-SFT 데이터셋을 구성한다.
```

## 10. 결론

LoRA/QLoRA 기반 sLLM 파인튜닝 실험은 기술적으로 정상 수행 가능한 구조로 구성되었다. `prepare_sllm_dataset.py`를 통해 통합 지원 정보 데이터를 instruction tuning 형식으로 변환할 수 있었고, `train_lora.py`는 LoRA와 QLoRA 학습을 모두 지원하도록 작성되었다. 또한 `sllm_prompt_test_results.jsonl` 기준 5개 대표 프롬프트에서 모두 오류 없이 한국어 응답을 생성하였다.

다만 응답 품질 측면에서는 최신성 판단, 출처 정확성, 오래된 정책 배제, 반복 생성 억제에서 한계가 확인되었다. 이는 sLLM이 실시간 검색 context 없이 학습 패턴만으로 답변을 생성했기 때문에 발생한 문제로 해석된다.

따라서 본 프로젝트의 운영 서비스에는 기존 OpenAI 기반 RAG 파이프라인을 유지하는 것이 적절하다. sLLM은 운영 모델을 대체하기보다는 향후 RAG context를 입력으로 받는 경량 답변 생성기, 조건 추출기, 요약기, 답변 초안 생성기 등 보조 모듈로 활용하는 방향이 바람직하다.

최종적으로 본 실험은 sLLM을 메인 운영 모델로 채택하기 위한 결과라기보다, 평가 기준에서 요구하는 PEFT 기반 파인튜닝 코드 역량과 실험적 검증 과정을 보여주는 보조 산출물로 의미가 있다.
