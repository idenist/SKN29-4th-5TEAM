# LLM-as-a-Judge 평가 보고서

## 1. 평가 개요

본 평가는 청년정책 RAG 서비스의 답변 품질을 LLM-as-a-Judge 방식으로 점검한 결과를 정리한 문서이다.  
규칙 기반 평가가 `route`, `next_action`, `recommendations`, 마감 공고 제외 여부처럼 시스템 동작 조건을 자동 점검하는 평가라면, LLM-as-a-Judge 평가는 검색된 컨텍스트와 생성 답변을 함께 보고 답변 품질을 정성적으로 점수화하는 평가이다.

이번 평가는 10개 질문을 대상으로 수행되었으며, 각 질문에 대해 다음 세 가지 지표를 1점부터 5점까지 채점하였다.

| 지표 | 의미 |
|---|---|
| Context Relevance | 검색된 정책·공고·훈련 과정이 사용자 질문과 얼마나 관련 있는지 |
| Groundedness | 생성된 답변이 검색된 정책 데이터에 얼마나 충실하게 근거하는지 |
| Answer Relevance | 생성된 답변이 사용자 질문에 얼마나 적절하게 답하는지 |

---

## 2. 평가 스크립트의 점수 산정 기준

평가 스크립트 `evaluate_rag_llm_judge.py`는 다음 흐름으로 동작한다.

```text
evaluation_dataset.jsonl에서 평가 질문 로드
→ random.seed(42) 기준으로 10개 샘플 선택
→ FastAPI /api/chat 엔드포인트 호출
→ RAG 응답의 recommendations를 검색 컨텍스트로 변환
→ RAG 답변을 LLM Judge 프롬프트에 입력
→ context_relevance, groundedness, answer_relevance를 1~5점으로 채점
→ llm_judge_results.json 저장
```

### 2.1 RAG 호출 방식

스크립트는 각 질문에 대해 다음 payload로 FastAPI 채팅 API를 호출한다.

```python
payload = {
    "message": question,
    "top_k": 3
}
```

즉, 각 질문마다 상위 3개 추천 후보를 중심으로 답변 품질을 평가한다.

### 2.2 Judge 입력 컨텍스트 구성

검색 컨텍스트는 RAG 응답의 `recommendations`에서 다음 필드를 가져와 구성한다.

```text
정책명(policy_name)
source_category
지원내용(support_content)의 앞 100자
```

따라서 Judge는 전체 원문 문서가 아니라, 추천 후보 목록과 요약 정보에 기반하여 판단한다. 이 때문에 검색 후보의 제목은 적절하지만 지원내용이 비어 있거나 짧은 경우 Groundedness가 보수적으로 평가될 수 있다.

### 2.3 답변 입력 범위

생성 답변은 전체 답변 중 앞 1,000자까지만 Judge에 전달된다.

```python
answer[:1000]
```

따라서 답변 후반부에 신청 URL, 유의사항, 추가 조건이 있더라도 Judge 평가에 충분히 반영되지 않을 수 있다.

### 2.4 Judge 모델 및 평가 지표

스크립트는 `gpt-4o-mini`를 Judge 모델로 사용하고, temperature는 0으로 설정한다. Judge는 다음 세 지표를 JSON 형식으로 반환한다.

```json
{
  "context_relevance": 1~5,
  "groundedness": 1~5,
  "answer_relevance": 1~5
}
```

각 지표의 의미는 다음과 같다.

| 지표 | 1점 | 3점 | 5점 |
|---|---|---|---|
| Context Relevance | 검색 결과가 질문과 거의 무관함 | 일부 관련은 있으나 핵심 의도와 약간 다름 | 검색 결과가 질문과 직접적으로 관련됨 |
| Groundedness | 답변이 검색 근거 없이 추정 또는 환각에 가까움 | 일부는 근거가 있으나 세부 정보가 부족함 | 답변이 검색된 정책 정보에 충분히 근거함 |
| Answer Relevance | 질문에 제대로 답하지 못함 | 질문에 일부 답하지만 불완전함 | 질문에 명확하고 충분히 답함 |

### 2.5 평가 스크립트의 한계

이번 스크립트는 품질 평가에 유용하지만 다음 한계가 있다.

```text
1. answer_item_ids 정답 ID와 실제 추천 ID를 직접 비교하지 않는다.
2. 검색 컨텍스트가 recommendations의 일부 필드로만 구성된다.
3. 답변은 앞 1,000자만 Judge에 전달된다.
4. LLM Judge 특성상 평가 모델에 따라 점수가 달라질 수 있다.
5. 10개 샘플 평가이므로 전체 질의 분포를 대표한다고 보기는 어렵다.
```

따라서 이 평가는 최종 정답률이라기보다, 검색 후보와 답변 품질의 정성적 안정성을 확인하는 보조 평가로 해석하는 것이 적절하다.

---


### 2.6 Rule-based 평가와의 차이

LLM-as-a-Judge 평가는 규칙 기반 평가와 목적이 다르다. 규칙 기반 평가는 route, next_action, recommendations 개수, 입력 검증, tool_trace 존재 여부처럼 시스템 동작 조건을 자동 점검한다. 반면 본 평가는 검색된 후보와 답변을 함께 보고 의미적 품질을 판단한다.

이번 Judge 스크립트는 `answer_item_ids`를 결과에 함께 저장하지만, 실제 추천 ID와 정답 ID의 Top-k Hit 여부를 직접 계산하지는 않는다. 따라서 본 결과는 정답률이라기보다 검색 후보와 답변의 관련성, 근거성, 응답 적절성을 보는 보조 품질 평가로 해석한다.

## 3. 전체 평가 결과 요약

| 지표 | 평균 점수 |
|---|---:|
| Context Relevance | 4.6 / 5 |
| Groundedness | 4.2 / 5 |
| Answer Relevance | 4.1 / 5 |
| 전체 평균 | 약 4.3 / 5 |

### 3.1 핵심 해석

전체적으로 세 지표가 모두 4점 이상으로 나타났다. 이는 검색된 컨텍스트가 대부분 질문과 관련 있고, 생성 답변도 대체로 검색 결과에 근거하여 작성되었음을 의미한다.

특히 Context Relevance 평균이 가장 높게 나타났는데, 이는 VectorDB 검색과 공식 외부 검색 fallback이 질문에 맞는 후보를 찾는 데 효과적으로 작동했음을 보여준다.

반면 Answer Relevance 평균은 상대적으로 낮다. 이는 대부분의 답변이 질문에 답은 하고 있으나, 실제 신청 가능 여부, 모집 기간, 자부담금, 제출 서류, 지역 제한 등 사용자가 바로 행동하기 위해 필요한 세부 정보가 부족했기 때문이다.

## 5. 각 TC 질문에 대한 답변 해석

### LLM-TC001. 지게차 자격증 국비지원 과정 있어?

| 항목 | 내용 |
|---|---|
| Route | 교육 |
| 주요 검색 결과 | 지게차 운전 입문자를 위한 자격증 취득 및 취업지원 / 지게차 운전자격증 취득과정(필기+실기) / 지게차운전기능사 자격증취득과정(실기) |
| Context Relevance | 5 / 5 |
| Groundedness | 4 / 5 |
| Answer Relevance | 4 / 5 |

**해석**  
질문 핵심인 지게차·자격증·국비지원과 직접 연결되는 교육훈련 과정이 검색되었다. 검색 적합성은 매우 높지만, 답변에서 신청 방법·제출 서류·자부담금 등 세부 신청 정보가 부족해 Groundedness와 Answer Relevance는 4점으로 평가되었다.

**개선 포인트**  
교육훈련 상세 페이지의 구조화 필드 추출을 강화하면 5점에 가까워질 수 있다.

### LLM-TC002. 청년 공공기관 인턴 기회 있어?

| 항목 | 내용 |
|---|---|
| Route | 일자리 |
| 주요 검색 결과 | 취업성공 디딤돌 청년인턴 사업 / 청년인턴 사업 / 삼척형 청년인턴 지원사업 |
| Context Relevance | 5 / 5 |
| Groundedness | 4 / 5 |
| Answer Relevance | 4 / 5 |

**해석**  
청년 인턴 기회와 직접 관련된 정책들이 검색되었고, 특히 공공기관 행정인턴 성격의 사업을 포함하여 질문 의도에 잘 맞는다. 다만 지역 조건, 모집 시기, 신청 절차 등 구체 정보가 충분히 구조화되지 않아 답변 완성도는 4점 수준이다.

**개선 포인트**  
지역 제한과 신청기간을 답변 상단에 요약하면 사용자 만족도가 더 높아진다.

| 주요 검색 결과 | "하루 한 끼 든든하게" 우본 자립준비청년 식비 지원 / [HWP] 서울시 자료 / 푸드뱅크 농식품 기부 기사 |
| Context Relevance | 4 / 5 |
| Groundedness | 4 / 5 |
| Answer Relevance | 4 / 5 |

**해석**

**개선 포인트**  
복지문화 분야의 외부 검색 결과는 본문 파싱과 문서형 결과 후처리가 필요하다.

**해석**

| Context Relevance | 4 / 5 |

**해석**

**개선 포인트**  
‘공제’, ‘내일채움공제’, ‘장기재직’ 키워드에 대한 exact match 또는 외부 공식 검색 보강이 필요하다.

## 6. 평가 결과 해석

### 6.1 검색 적합성은 전반적으로 높다

Context Relevance 평균은 4.6/5로 가장 높았다. 이는 사용자의 질문을 도메인별로 라우팅한 뒤, 내부 VectorDB와 공식 외부 검색 fallback을 통해 관련 후보를 상당히 잘 찾아오고 있음을 의미한다.

특히 다음 케이스는 검색 결과가 질문과 직접적으로 연결되었다.

```text
- 지게차 자격증 국비지원 과정
- 청년 공공기관 인턴
- 청년 직무 일경험
- 산림 기반 창업 비즈니스 모델
- 중소기업 청년 교통비 지원
```

이 결과는 초기 RAG의 문제였던 “관련 없는 내부 후보를 억지 추천하는 현상”이 상당 부분 개선되었음을 보여준다.

### 6.2 Groundedness는 안정적이지만 세부 정보 부족이 남아 있다

Groundedness 평균은 4.2/5로, 답변이 대체로 검색된 정책과 공고에 근거하고 있음을 보여준다.

다만 대부분의 케이스에서 5점이 아니라 4점이 나온 이유는 다음과 같다.

```text
- 검색 결과 title/snippet 중심으로 답변한 외부 검색 케이스가 있음
- 신청기간, 신청방법, 제출서류가 명확하지 않은 후보가 있음
- HWP/PDF/문서뷰어 결과가 일부 포함됨
- 지원내용 필드가 비어 있거나 짧은 경우가 있음
```

즉, 환각이 심한 상태는 아니지만, 상세 페이지 본문을 구조화하지 못한 한계가 점수에 반영되었다.

### 6.3 Answer Relevance는 “답은 하지만 아직 신청 안내로는 부족한” 수준이다

Answer Relevance 평균은 4.1/5로 세 지표 중 가장 낮다. 그러나 모든 케이스가 4점 이상이므로, 질문과 무관한 답변을 하는 수준은 아니다.

점수가 4점에 머문 주요 원인은 다음과 같다.

```text
- “있어?”라는 질문에는 답하지만 바로 신청 가능한지 확정하지 못함
- 세부 자격 조건이 “추가 확인 필요”로 남는 경우가 많음
- 외부 검색 결과는 공식 출처 확인 후보로 안내되지만, 실제 신청 절차까지 확정하지 못함
- 일부 질문은 정확한 제도명과 유사 제도가 섞임
```

대표적으로 “중소기업 재직 청년 장기 재직 지원 공제” 질문에서는 중소기업 재직 청년과 관련된 후보가 검색되었지만, 정확한 공제 제도와 완전히 일치하지는 않아 4점으로 평가되었다.

### 6.4 공식 외부 검색 fallback의 효과가 확인되었다

다음 케이스들은 내부 VectorDB만으로는 충분하지 않을 수 있는 롱테일 질문에 해당한다.

```text
- 취약계층 청년 식품 지원
- 청년 고용 정책 행사
- 청년 창업자 임차료 지원
- 중소기업 청년 교통비 지원
```

이들 케이스에서 공식 외부 출처 검색 결과가 활용되었고, 대부분 4점 이상을 받았다. 이는 Tavily 기반 공식 도메인 제한 검색이 내부 VectorDB의 한계를 보완하는 fallback으로 효과가 있음을 보여준다.

다만 외부 검색 결과에는 HWP, 문서뷰어, 일반 공지 페이지가 섞일 수 있으므로 후처리와 rerank는 계속 필요하다. 또한 외부 검색이 `planned_not_executed`로 기록되는 경우는 단순히 분기하지 않았다는 뜻이 아니라, 공식 출처 검색 또는 검색 계획 단계까지 갔으나 후처리 필터를 통과한 사용 가능 결과가 없었다는 의미로 해석해야 한다.

## 8. 추가 개선 방향

### 8.1 공식 상세 페이지 본문 파싱

외부 검색 결과의 title/snippet만 사용하는 방식에서 벗어나, 공식 상세 페이지 본문을 가져와 다음 필드를 구조화하면 Groundedness와 Answer Relevance를 함께 높일 수 있다.

```text
- 신청 기간
- 지원 대상
- 지역 제한
- 지원 금액
- 신청 방법
- 제출 서류
- 담당 기관
```

### 8.2 HWP/PDF/문서뷰어 결과 후처리 강화

현재 일부 케이스에서 HWP 또는 문서뷰어 결과가 검색 후보에 포함된다. 이러한 결과는 공식성이 있을 수 있지만, snippet만으로 답변 근거를 만들기 어렵다.

개선 방향은 다음과 같다.

```text
- HWP/PDF는 기본적으로 후순위 처리
- 동일한 내용의 HTML 상세 페이지가 있으면 HTML 우선 사용
- 문서뷰어 제목만 있는 결과는 본문 추출 실패 시 제외
```

### 8.3 질문 핵심어 exact match 강화

“공제”, “임차료”, “행사”, “식품”처럼 사용자가 구체적으로 요구한 단어는 단순 관련 분야보다 더 높은 가중치를 줘야 한다.

예시:

```text
중소기업 재직 청년 장기 재직 지원 공제
→ 공제 / 장기재직 / 내일채움 계열 키워드 우선
→ 바우처, 응시료, 취업지원제도는 후순위
```

### 8.4 평가 스크립트 개선

현재 LLM Judge 스크립트는 품질 평가에 유용하지만, 더 정밀한 평가를 위해 다음 보완이 가능하다.

```text
1. answer_item_ids와 실제 추천 ID의 Top-k Hit 여부 계산
2. 같은 질문을 2~3회 Judge하여 평균 점수 사용
3. Judge 입력에 full chunk 또는 source_url 포함
4. 외부 검색 결과와 내부 VectorDB 결과를 구분하여 별도 평가
5. context_relevance, groundedness, answer_relevance 외에 freshness_safety와 react_trace_quality 추가
```

## 9. 보고서용 요약 문장

```text
LLM-as-a-Judge 평가 결과, 10개 테스트 질문에서 Context Relevance 평균 4.6/5, Groundedness 평균 4.2/5, Answer Relevance 평균 4.1/5를 기록하였다. 이는 검색된 정책 후보가 사용자 질문과 대체로 관련성이 높고, 생성 답변 또한 검색 결과에 근거하여 작성되고 있음을 보여준다.

특히 지게차 자격증, 청년 공공기관 인턴, 청년 직무 일경험, 산림 기반 창업 프로그램, 중소기업 청년 교통비 지원과 같은 질의에서는 질문과 직접 관련된 후보가 검색되었다. 또한 내부 VectorDB에서 부족한 롱테일 질의에 대해서는 공식 도메인 제한 외부 검색 fallback을 통해 보완하는 구조가 효과적으로 작동하였다.

다만 대부분의 Answer Relevance가 4점에 머문 것은 신청 가능 여부, 모집 기간, 제출 서류, 지역 제한 등 세부 신청 정보가 충분히 구조화되지 않았기 때문이다. 따라서 향후에는 공식 상세 페이지 본문 파싱, HWP/PDF 후처리, 질문 핵심어 기반 rerank를 강화하여 사용자가 바로 신청 판단을 할 수 있는 수준으로 답변 품질을 개선할 필요가 있다.
```

## 10. 결론

이번 LLM-as-a-Judge 평가 결과는 현재 RAG 시스템이 단순히 문장을 생성하는 수준을 넘어, 검색 후보의 관련성과 답변의 근거성을 일정 수준 이상 확보하고 있음을 보여준다.

최종적으로 다음과 같이 해석할 수 있다.

```text
현재 시스템은 검색 적합성 측면에서 안정적인 수준에 도달했다.
답변도 대체로 검색 결과에 근거하고 있어 환각 위험은 낮아졌다.
다만 신청 가능 여부를 확정할 수 있는 세부 구조화 정보가 부족하여,
정확한 신청 안내 서비스로 발전하려면 상세 페이지 본문 파싱과 조건 추출 고도화가 필요하다.
또한 Graph DB 보조 검색 결과를 실제 답변 후보로 더 적극 활용하려면 Graph 노드에 source_url, application_url, summary 같은 상세 필드를 함께 저장하고 반환하도록 개선해야 한다.
```
