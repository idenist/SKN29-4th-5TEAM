# 백엔드 / RAG 연결 문서 최종본

## 0. 문서 목적

본 문서는 데이터 전처리 결과물, FastAPI 백엔드, Chroma 기반 RAG 검색 파이프라인, Streamlit 프론트엔드가 같은 key와 필드 의미를 기준으로 연동되도록 정리한 전달 문서이다.

이번 최종본에는 다음 수정 사항을 반영하였다.

| 구분 | 반영 여부 | 연결 관점의 의미 |
|---|---:|---|
| 1순위 | 반영 | RAG 검색 결과에서 `application_url`이 사라지지 않고 `/api/chat` 추천 카드까지 전달된다. |
| 2순위 | 반영 | 통합 기준 필드 `item_id/title`과 기존 호환 필드 `policy_id/policy_name`을 함께 유지한다. |
| 3순위 | 코드 미수정, 문서 반영 | `/api/policies`는 현재 코드 기준인 `limit/offset` pagination을 사용한다. |
| 4순위 | 반영 | 마감 상태는 `startup_notice` 중심으로 계산하며, 그 외 유형은 기본 `unknown`, `false`로 처리한다. |
| 5순위 | 미반영 | `region_code`, `keywords`는 현재 필수 API 계약에 포함하지 않는다. |

핵심 연결 원칙은 다음과 같다.

```text
opportunity_chunks.jsonl에서 검색한다.
검색 결과의 item_id로 opportunities.json 상세 데이터와 연결한다.
API와 프론트에서는 item_id/title을 우선 사용한다.
기존 코드 호환을 위해 policy_id/policy_name alias를 함께 유지한다.
application_url은 추천 카드의 신청 버튼 표시를 위해 반드시 보존한다.
```

## 4. 핵심 연결 key

### 4.1 `item_id` 사용 원칙

`item_id`는 백엔드, RAG, 프론트엔드가 공통으로 사용하는 단일 식별자이다.

| 사용 위치 | 사용 방식 |
|---|---|
| `opportunities.json` | 상세 데이터의 primary key |
| `opportunity_chunks.jsonl` | chunk metadata에 저장 |
| Chroma metadata | 검색 결과에서 원본 상세 데이터 연결용 |
| `/api/policies/{item_id}` | 상세 조회 path key |
| `/api/chat recommendations[]` | 추천 카드, 상세 이동, 로그 추적 key |
| Streamlit 추천 카드 | 상세 페이지 이동, 버튼 클릭, 중복 관리 key |

### 4.2 `title` 사용 주의사항

`title`은 화면 표시용 이름일 뿐 고유 key가 아니다.

```text
동일한 교육훈련 과정이 회차별로 여러 번 존재할 수 있다.
동일한 창업공고 제목이 연도 또는 기관별로 반복될 수 있다.
따라서 title을 URL path, DB primary key, 상세 조회 key로 사용하면 안 된다.
```

### 4.3 기존 코드 호환 alias

기존 코드가 정책 중심으로 작성되어 있으므로 alias를 유지한다.

| 통합 기준 필드 | 기존 호환 필드 | 유지 방식 |
|---|---|---|
| `item_id` | `policy_id` | 같은 값으로 내려준다. |
| `title` | `policy_name` | 같은 값으로 내려준다. |
| `source_category` | 신규 필드 | 반드시 보존한다. |
| `content` 또는 `text` | LLM context | Answer Generator의 근거 문맥으로 활용한다. |

## 5. 백엔드 담당자 전달 사항

백엔드는 다음 파일을 기본 데이터로 사용한다.

```text
data/processed/opportunities.json
```

### 5.1 백엔드 구현 규칙

| 규칙 | 설명 |
|---|---|
| 상세 조회 key | 반드시 `item_id` 기준으로 조회한다. |
| 목록 조회 | `source_category`, `domain`, `keyword`, `limit`, `offset`을 지원한다. |
| pagination | 현재 코드는 `page/page_size`가 아니라 `limit/offset` 방식이다. |
| 신청 버튼 | `application_url`이 없으면 버튼을 숨긴다. |
| 출처 버튼 | `source_url`이 있으면 출처 보기 링크로 표시한다. |
| 데이터 완성도 | `info_score`는 추천 점수가 아니라 데이터 완성도 점수다. |
| 상세 확인 | `needs_detail_check=true`이면 원문 확인 문구를 표시한다. |
| 결측치 | 원천 데이터의 빈 문자열은 API 또는 프론트에서 `null`처럼 처리한다. |
| 원본 미포함 정보 | 신청방법, 제출서류, 자격조건을 임의로 생성하지 않는다. |

### 5.2 `/api/policies` pagination 기준

3순위 코드 수정은 진행하지 않았으므로 문서를 현재 코드에 맞춰 `limit/offset` 방식으로 정리한다.

```http
GET /api/policies?source_category=policy&keyword=청년수당&limit=20&offset=0
```

프론트에서 페이지 번호를 쓰고 싶다면 다음과 같이 변환한다.

```text
limit = page_size
offset = (page - 1) * page_size
```

예시:

| 페이지 | limit | offset |
|---:|---:|---:|
| 1페이지 | 20 | 0 |
| 2페이지 | 20 | 20 |
| 3페이지 | 20 | 40 |

### 5.3 백엔드 응답 필드 권장안

목록과 상세 조회에서는 `item_id/title`을 우선 내려주고, 기존 코드 호환을 위해 `policy_id/policy_name`도 함께 내려준다.

```json
{
  "item_id": "policy_20260605005400113228",
  "title": "청년미래적금",
  "policy_id": "policy_20260605005400113228",
  "policy_name": "청년미래적금",
  "source_category": "policy",
  "domain": "금융",
  "summary": "청년의 자산 형성을 지원하는 적금 상품",
  "source_url": "https://...",
  "application_url": "https://...",
  "info_score": 90,
  "needs_detail_check": true
}
```

### 5.4 빈 값 처리 기준

| 원천 값 | API 응답 권장 | 프론트 처리 |
|---|---|---|
| `""` | `null` 또는 빈 문자열 유지 | 영역 숨김 |
| `null` | `null` | 영역 숨김 |
| 값 존재 | 값 그대로 반환 | 카드/상세 화면 표시 |
| `needs_detail_check=true` | `true` | `원문 확인 필요` 문구 표시 |

> 백엔드에서 `정보 없음` 문자열을 넣으면 프론트에서 결측 영역을 숨기기 어려워진다. 가능하면 API에서는 `null` 또는 빈 문자열을 유지하고, 화면에서만 필요 시 `정보 없음`을 표시한다.

## 6. RAG 담당자 전달 사항

RAG는 다음 파일을 사용한다.

```text
data/processed/opportunity_chunks.jsonl
```

### 6.1 chunk 구조

권장 chunk 구조는 다음과 같다.

```json
{
  "chunk_id": "training_0001_section_summary",
  "item_id": "training_0001",
  "source_category": "training",
  "domain": "education",
  "title": "K-Digital Training 데이터 분석 과정",
  "content": "훈련과정명: ...\n훈련기관: ...\n훈련기간: ...",
  "metadata": {
    "item_id": "training_0001",
    "title": "K-Digital Training 데이터 분석 과정",
    "policy_id": "training_0001",
    "policy_name": "K-Digital Training 데이터 분석 과정",
    "source_category": "training",
    "domain": "education",
    "section": "summary",
    "source_url": "https://...",
    "application_url": "https://...",
    "needs_detail_check": false,
    "info_score": 85
  }
}
```

### 6.2 임베딩 규칙

| 항목 | 규칙 |
|---|---|
| 임베딩 대상 | `content`만 임베딩 본문으로 사용한다. |
| metadata 저장 | `item_id`, `title`, `policy_id`, `policy_name`, `source_category`, `domain`, `section`, `source_url`, `application_url`, `needs_detail_check`, `info_score`를 metadata로 저장한다. |
| 원본 연결 | 검색 결과는 `item_id` 기준으로 `opportunities.json` 상세 데이터와 연결한다. |
| 중복 제거 | chunk 검색 후 `item_id` 기준으로 중복 제거한다. |
| 답변 생성 | 검색된 context와 원본 필드에 없는 내용은 생성하지 않는다. |
| 상세 확인 | `needs_detail_check=true`이면 답변에서 원문 확인을 유도한다. |

### 6.3 Retriever 후처리 권장 순서

```text
1. 사용자 query semantic search
2. age/region 등 구조화 필터 적용
3. source_category 우선 필터 적용
4. domain 우선 필터 적용
5. item_id 기준 중복 제거
6. startup_notice 마감 상태 계산
7. score/info_score/needs_detail_check 기준 보조 정렬
8. top_k 개수만큼 최종 반환
```

### 6.4 `/api/chat` 추천 결과로 보존해야 하는 필드

RAG 검색 결과에서 FastAPI 응답까지 최소한 다음 필드는 보존되어야 한다.

| 필드 | 이유 |
|---|---|
| `item_id` | 상세 조회, 중복 제거, 프론트 key |
| `title` | 카드 제목 |
| `policy_id` | 기존 코드 호환 |
| `policy_name` | 기존 코드 호환 |
| `source_category` | 뱃지, 필터링 |
| `domain` | 분야 표시 |
| `summary` | 카드 요약 |
| `source_url` | 출처 보기 버튼 |
| `application_url` | 신청 버튼 |
| `needs_detail_check` | 원문 확인 필요 표시 |
| `deadline_status` | 마감 필터 |
| `application_end_date` | 마감일 표시 |
| `is_expired` | 마감 여부 |

### 6.5 source_category fallback 주의

source_category를 강하게 필터링하면 검색 결과가 비는 문제가 생길 수 있다. 다만 너무 느슨하게 fallback하면 사용자가 교육훈련을 물었는데 정책이 섞이는 문제가 생긴다.

권장 정책:

```text
1. 지정 source_category 결과가 1개 이상 있으면 해당 category 결과를 우선 유지한다.
2. 지정 source_category 결과가 전혀 없을 때만 전체 category fallback을 수행한다.
3. fallback이 발생하면 warnings에 "일부 결과는 요청한 유형과 다를 수 있음"을 추가한다.
```

## 7. 검색 결과 표시 제안

training 데이터가 전체 데이터에서 차지하는 비중이 크므로, 전체 검색에서는 source_category별 균형 노출이 필요하다.

### 7.1 전체 검색 균형 노출 예시

| 유형 | 노출 개수 예시 |
|---|---|
| 청년정책 | 3개 |
| 창업지원 | 3개 |
| 교육·훈련 | 3개 |

### 7.2 top_k 해석 기준

현재 API 명세에서는 `top_k`를 최종 반환 개수 기준으로 본다. 전체 검색에서 균형 노출이 필요하면 내부적으로 category별 후보를 더 많이 가져온 뒤 최종 `top_k`개를 재정렬한다.

| 방식 | 설명 | 장점 | 주의점 |
|---|---|---|---|
| 최종 개수 기준 | category별 후보를 넓게 검색한 뒤 최종 `top_k`개 반환 | 프론트 처리 단순 | category 균형이 약해질 수 있음 |
| category별 개수 기준 | category별 `top_k`개씩 반환 | 균형 노출 쉬움 | 최종 개수가 `top_k`보다 커질 수 있음 |

## 8. 마감 상태 처리

창업지원 공고는 오래된 마감 공고가 검색될 가능성이 크므로 `deadline_status`, `application_end_date`, `is_expired`를 함께 제공한다.

| 필드 | 값 | 설명 |
|---|---|---|
| `deadline_status` | `open` | 아직 마감 전 |
| `deadline_status` | `expired` | 이미 마감됨 |
| `deadline_status` | `unknown` | 마감일 확인 불가 또는 마감 판단 대상 아님 |
| `application_end_date` | `YYYY-MM-DD` 또는 `null` | 파싱된 신청/접수 종료일 |
| `is_expired` | `true/false` | 마감 여부 |

### 8.1 source_category별 마감 처리 기준

| source_category | 처리 기준 |
|---|---|
| `startup_notice` | 신청/접수 종료일을 파싱하여 마감 상태 계산 |
| `policy` | 기본 `deadline_status="unknown"`, `is_expired=false` |
| `training` | 기본 `deadline_status="unknown"`, `is_expired=false` |

### 8.2 마감 필터링 기준

4순위 수정에 따라 strict freshness 필터는 `startup_notice`에 우선 적용한다.

```text
- "지금 신청 가능한 창업공고", "모집 중인 예비창업자 사업화 지원" 등은 expired 결과를 강하게 제외한다.
- policy/training은 종료일 의미가 데이터마다 다를 수 있으므로 기본적으로 expired 필터를 강하게 적용하지 않는다.
- policy/training의 마감 처리를 강화하려면 별도의 application_end_date 정제 로직을 추가한 뒤 적용한다.
```

### 8.3 프론트 필터링 예시

```python
show_expired = st.checkbox("마감된 공고 포함", value=False)

if not show_expired:
    recommendations = [
        item for item in recommendations
        if item.get("deadline_status") != "expired"
    ]
```

## 9. Ground Truth 평가 데이터셋

RAG 검색 품질 검증을 위해 아래 파일을 사용한다.

```text
tests/evaluation_dataset.jsonl
```

각 row는 사용자 질문 `question`과 정답으로 기대되는 `answer_item_ids`를 포함한다.

예상 구조:

```json
{
  "question": "서울 청년 월세 지원 정책 알려줘",
  "answer_item_ids": ["policy_..."],
  "source_category": "policy",
  "memo": "주거 정책 검색 평가"
}
```

검증 스크립트:

```bash
python scripts/validate_evaluation_dataset.py
```

검증 기준:

| 항목 | 기준 |
|---|---|
| ID 존재성 | `answer_item_ids`가 `opportunities.json`에 존재해야 함 |
| 검색 적합성 | Top-k 결과에 정답 `item_id`가 포함되는지 확인 |
| category 라우팅 | 질문 의도와 `source_category`가 일치하는지 확인 |
| 답변 근거성 | 생성 답변이 검색 context와 원본 필드에 근거하는지 확인 |

## 12. 위험 요소 및 대응

| 위험 요소 | 영향 | 대응 |
|---|---|---|
| `item_id`와 `policy_id` 혼용 | 상세 조회 실패, 프론트 key 오류 | alias를 함께 내려주고 최종 기준은 `item_id`로 통일 |
| `title` 중복 | 잘못된 상세 페이지 이동 | title을 key로 사용 금지 |
| `application_url` 누락 | 신청 버튼 표시 불가 | RAG 응답과 목록/상세 API에 모두 포함 |
| `/api/policies` pagination 혼선 | 프론트 호출 실패 | 문서를 `limit/offset` 기준으로 통일 |
| `info_score` 오해 | 사용자 적합도 점수처럼 잘못 표시 | 데이터 완성도 점수라고 명시 |
| `needs_detail_check` 미표시 | 불완전 정보가 확정 정보처럼 보임 | 원문 확인 필요 문구 표시 |
| training 결과 과다 노출 | 정책/창업공고가 묻힘 | source_category 균형 노출 또는 라우팅 가중치 적용 |
| 마감 공고 상위 노출 | 사용자 불만 및 시연 품질 저하 | `deadline_status` 계산 후 expired 기본 숨김 |
| 원본 미포함 정보 생성 | 환각 및 잘못된 안내 | 신청방법/서류/조건은 원본 필드 기반으로만 출력 |

## 14. 최종 정리

백엔드/RAG 연결에서 가장 중요한 점은 `item_id`를 기준으로 검색 결과와 원본 상세 데이터를 안정적으로 연결하는 것이다. RAG는 `opportunity_chunks.jsonl`의 `content`를 임베딩하고, 검색 결과 metadata에 포함된 `item_id`로 `opportunities.json`의 상세 데이터를 참조한다.

프론트엔드는 `source_category`로 뱃지를 표시하고, `application_url` 유무로 신청 버튼을 제어하며, `needs_detail_check`와 `deadline_status`를 활용해 원문 확인 필요 여부와 마감 여부를 명확히 안내한다.

최종적으로 다음 흐름이 유지되면 된다.

```text
사용자 질문
→ route/source_category 판단
→ Chroma 검색
→ item_id 기준 중복 제거 및 상세 데이터 연결
→ 자격 가능성 판단
→ application_url/source_url 보존
→ 출처 기반 답변 생성
→ Streamlit 추천 카드 표시
```
