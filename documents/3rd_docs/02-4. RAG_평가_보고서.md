# RAG 평가 보고서

## 1. 평가 개요

본 보고서는 `evaluation_results.md`와 `evaluation_results.json` 산출물을 기반으로, 청년정책 RAG 서비스의 규칙 기반 평가 결과와 케이스별 응답 품질을 해석한 문서이다. 평가는 총 10개의 테스트 케이스로 구성되었으며, 정상 추천, 마감 공고 배제, 비현실적 질의 fallback, 짧은 질의 처리, 빈 질의 검증 등을 포함한다.

| 항목 | 결과 |
|---|---:|
| 총 테스트 케이스 수 | 10 |
| 통과 케이스 수 | 10 |
| 통과율 | 100.0% |
| 평균 규칙 기반 점수 | 1.000 |
| 평균 BLEU | 0.0212 |
| 평균 ROUGE-1 F1 | 0.1451 |
| 평균 ROUGE-2 F1 | 0.0581 |
| 평균 ROUGE-L F1 | 0.1242 |

요약하면, 규칙 기반 평가에서는 10개 케이스가 모두 통과하여 100% 통과율을 달성하였다. 다만 BLEU/ROUGE는 추천형 RAG 특성상 참고 지표로 해석해야 하며, 실제 품질 평가는 추천 후보의 적합성, 마감 공고 배제, ReAct trace, fallback 동작을 함께 확인해야 한다.

## 2. 평가 스크립트의 점수 산정 기준

평가 스크립트는 각 테스트 케이스에 대해 FastAPI 채팅 API를 호출한 뒤, 응답 JSON을 기준으로 여러 개의 규칙 기반 check를 수행한다. 각 check가 `True`이면 통과, `False`이면 실패로 기록된다.

### 2.1 규칙 기반 점수

각 케이스의 규칙 기반 점수는 다음 방식으로 계산된다.

```text
Rule Score = 통과한 check 수 / 전체 check 수
Case Pass = 모든 check가 True일 때만 Pass
```

현재 평가에서 사용된 주요 check는 다음과 같다.

| 평가 항목 | 의미 |
|---|---|
| `must_have_any` | 응답에 기대 키워드 또는 기대 문구가 포함되는지 확인 |
| `must_not_have_any` | 금지 후보가 추천 결과에 포함되지 않았는지 확인 |
| `route` | 질문이 기대한 정책 분야로 라우팅되었는지 확인 |
| `internal_search_sufficient` | 내부 VectorDB 검색 충분성 판단 결과가 기대와 맞는지 확인 |
| `next_action` | `answer_generation` 또는 `external_search` 분기가 기대와 맞는지 확인 |
| `external_used` | 외부 검색 사용 여부가 기대와 맞는지 확인 |
| `external_search_status` | 외부 검색 상태가 기대와 맞는지 확인 |
| `external_search_targets` | 고용24/온통청년/K-Startup 등 fallback 대상이 적절한지 확인 |
| `http_status` | 정상 요청 또는 입력 검증 실패 응답이 기대 범위인지 확인 |
| `min_recommendations`, `max_recommendations` | 추천 개수가 기대 범위에 있는지 확인 |
| `no_expired_recommendations` | 마감된 추천이 포함되지 않았는지 확인 |
| `expected_warning_any` | 필요한 경고 메시지가 포함되는지 확인 |
| `expected_error_any` | 입력 오류 등 기대 오류가 포함되는지 확인 |
| `tool_trace_present_when_needed` | ReAct 흐름을 설명할 수 있는 `tool_trace`가 존재하는지 확인 |

특히 `must_not_have_any`는 전체 답변 본문을 무조건 검사하지 않고, 기본적으로 `recommendations`를 중심으로 검사한다. 이는 비현실적 질문에서 “마감 후보로 제외됨”처럼 설명 목적으로 후보명을 언급한 경우를 잘못 실패 처리하지 않기 위한 조정이다.

### 2.2 BLEU/ROUGE 지표

평가 스크립트는 `reference_answer` 또는 `expected_answer`가 있는 경우 BLEU와 ROUGE를 계산한다. 하지만 본 서비스는 정답 문장을 그대로 생성하는 번역/요약 시스템이 아니라, 검색 결과를 기반으로 정책을 추천하고 구조화해 설명하는 RAG 시스템이다. 따라서 같은 정책을 정확히 추천하더라도 문장 형식, 항목 순서, 주의사항 표현이 다르면 BLEU/ROUGE는 낮게 나올 수 있다.

따라서 BLEU/ROUGE는 “참고용 텍스트 유사도”로 보고, 최종 성능 해석은 규칙 기반 점수와 LLM-as-a-Judge 평가, 그리고 실제 추천 후보의 품질을 함께 보는 것이 적절하다.


### 2.3 평가셋과 평가 목적 구분

본 규칙 기반 평가는 `query`, `expected_route`, `expected_next_action`, `must_have_any`, `must_not_have_any` 등 자동 검증 조건을 포함한 평가셋을 사용한다. 반면 LLM-as-a-Judge 평가는 `question`, `answer_item_ids` 중심의 샘플 평가셋을 사용하여 검색 후보와 생성 답변의 의미 품질을 판단한다.

따라서 두 평가는 목적이 다르다.

| 평가 방식 | 주 목적 | 해석 방식 |
|---|---|---|
| Rule-based 평가 | route, fallback, 입력 검증, 마감 배제, tool_trace 등 시스템 동작 확인 | 기능 안정성 평가 |
| BLEU/ROUGE | 기준 답변과 생성 답변의 텍스트 유사도 참고 | 보조 지표 |
| LLM-as-a-Judge | 검색 후보와 답변의 의미적 관련성·근거성 평가 | 품질 보완 평가 |

규칙 기반 100% 통과는 “정해진 시스템 동작 조건을 모두 만족했다”는 의미이며, 모든 추천 후보가 의미적으로 완벽하다는 뜻은 아니다.

## 3. 전체 평가 결과 해석

최종 결과는 10개 테스트 케이스 모두 통과이며, 평균 규칙 기반 점수도 1.000으로 나타났다. 이는 다음 기능들이 자동 평가 기준상 정상적으로 동작했음을 의미한다.

- 질문 의도에 따른 분야 라우팅
- 내부 VectorDB 검색과 검색 충분성 판단
- 마감/오래된 공고 배제
- 비현실적 질문에 대한 추천 억제 및 fallback
- Tavily 기반 공식 도메인 외부 검색 fallback
- 짧은 질문 및 빈 질문에 대한 방어적 응답
- ReAct 흐름을 확인할 수 있는 `tool_trace` 기록

다만 규칙 기반 평가가 100%라고 해서 모든 답변이 의미적으로 완벽하다는 뜻은 아니다. 일부 케이스에서는 자동 check는 통과했지만, 실제 추천 후보 품질 관점에서 개선 여지가 남아 있다. 특히 `면접수당` 케이스는 질문과 직접 관련성이 낮은 정책이 추천되었고, `청년도약계좌` 케이스는 관련 정책이 포함되었지만 순위가 3번째로 밀렸다. 따라서 보고서에서는 “규칙 기반 안정성은 확보했으나 의미적 품질 개선은 추가 필요”로 해석하는 것이 가장 안전하다.

### TC001_startup_open_2026

- 질문: `2026년에 신청 가능한 창업지원 공고 추천해줘`
- Route: `창업`
- Next Action: `answer_generation`
- External Used: `False`
- 추천 수: `1`
- 주요 추천/후보: `2026년 강북창업지원센터 6월  창업 상담 참가자 모집(정부지원사업 사업계획서 작성 그룹 멘토링)`

창업 분야로 정확히 라우팅되었고, `startup_notice` source_category가 선택되었다. 내부 VectorDB 검색 결과는 1건이었으며, 해당 공고는 `deadline_status=open`, `is_expired=false`로 확인되었다. 따라서 외부 검색 없이 내부 RAG만으로 답변을 생성했다. 이 케이스는 최신 창업지원 공고 추천과 마감 공고 배제 로직이 정상 작동한 사례이다.

- Route: `창업`
- Next Action: `answer_generation`
- External Used: `False`

- 주요 추천/후보: `[창업] 내 아이디어로 창업 가능할까?`

질문에 “2019년에 했던 … 말고 지금 신청 가능한”이라는 현재성 의도가 포함되어 있었다. 시스템은 창업 route와 `startup_notice`를 선택했고, 과거 공고 대신 2026년 현재 신청 가능한 창업 공고를 추천했다. 이는 오래된 공고 배제와 현재성 질문 처리 로직이 정상적으로 동작했음을 보여준다.

### TC003_react_fallback_unrealistic

- 질문: `화성에서 받을 수 있는 청년 우주창업 지원금 알려줘`
- Route: `창업`
- Next Action: `external_search`
- External Used: `False`
- 추천 수: `0`
- 주요 추천/후보: 없음

“화성”, “우주창업”처럼 내부 데이터에서 현실적인 정책 후보를 찾기 어려운 질문이다. 내부 검색에서는 5건이 잡혔지만 모두 마감 후보였고, keyword overlap도 0으로 판단되어 모두 제거되었다. 이후 K-Startup 공식 출처 확인 대상으로 분기했지만 후처리 필터를 통과한 외부 결과가 없어 `recommendations=[]`로 응답했다. 이는 LLM이 없는 정책을 지어내지 않고 fallback 안내를 제공한 사례이다.

### TC004_housing_monthly_rent

- 질문: `서울에 사는 25세 청년이 받을 수 있는 월세 지원 정책 알려줘`
- Route: `주거`
- Next Action: `answer_generation`
- External Used: `False`
- 추천 수: `5`
- 주요 추천/후보: `(국토부) 26년 청년월세 지원사업`, `청년월세 한시 특별지원`, `청년안심주택 공급활성화(임차보증금 무이자지원)`

서울, 25세, 월세 조건을 바탕으로 주거 route가 선택되었다. 내부 VectorDB에서 청년월세 지원사업을 포함한 후보들이 검색되었고, 첫 번째 추천은 국토부 청년월세 지원사업으로 적절하다. 다만 후순위에 청년수당, 참여 네트워크 등 월세와 직접성이 낮은 후보가 일부 섞일 수 있어, 향후 top-k rerank 또는 direct intent filtering을 더 강화할 수 있다.

- Route: `교육`
- Next Action: `answer_generation`
- External Used: `False`
- 추천 수: `5`
- 주요 추천/후보: `[AI융복합과정]컴퓨터활용능력 2급 취득 및 AI 업무 활용 과정 (필기&실기)`, `Ai를 활용한 유튜브영상편집`, `Ai를 활용한 유튜브영상편집`

- 추천 수: `1`
- 주요 추천/후보: `모자건강증진 사업`

규칙 기반 평가는 통과했지만, 실제 응답 품질 관점에서는 가장 주의가 필요한 케이스이다. 질문은 면접수당 정책을 요구했으나 추천된 후보는 `모자건강증진 사업`으로, 면접수당과 직접 관련성이 낮다. 평가셋의 규칙 조건이 route와 최소 추천 수 중심이라 통과한 것으로 보이며, 향후 `must_have_policy_name_any`, `expected_primary_keyword_any` 같은 정밀 평가 조건을 추가해야 한다.

### TC007_finance_savings

- 질문: `청년도약계좌 같은 금융 정책 알려줘`
- Route: `금융`
- Next Action: `answer_generation`
- External Used: `False`
- 추천 수: `5`
- 주요 추천/후보: `청년미래적금('26.6.22 출시예정)`, `(접수 마감)2026년 11기 광주 청년 13(일+삶)통장`, `청년 자산형성 지원(청년도약계좌)`

금융 route는 적절하며 청년도약계좌가 추천 목록에 포함되었다. 다만 가장 직접적인 `청년 자산형성 지원(청년도약계좌)`가 3순위로 밀렸고, `접수 마감`이라는 제목이 포함된 후보가 2순위에 등장했다. `deadline_status`는 expired로 구조화되지 않아 no_expired check는 통과했지만, 제목 기반 마감 감지와 청년도약계좌 직접 매칭 rerank를 강화할 필요가 있다.

### TC008_participation_committee

- 질문: `청년 참여위원회 모집 정보 알려줘`
- Route: `참여권리`
- Next Action: `external_search`
- External Used: `False`
- 추천 수: `0`
- 주요 추천/후보: 없음

참여권리 route로 정상 분기했고, 내부 검색 결과 중 창업/창직/과거 해외연수 후보가 route/source_category 불일치로 제거되었다. 외부 검색도 시도했지만 후처리 필터를 통과한 결과가 없어 공식 출처 확인 안내로 응답했다. 실제 추천을 하지 않은 것은 안전하지만, 사용자 만족도 측면에서는 온통청년 또는 지자체 참여위원회 검색어 확장이 필요하다.

### TC009_short_query_warning

- 질문: `월세`
- Route: `주거`
- Next Action: `external_search`
- External Used: `True`
- 추천 수: `3`
- 주요 추천/후보: `청년정책 > 청년정책검색 > 중앙정부/타지역 정책 | 청년몽땅정보통`, `서울시, 청년 월세 지원 확대…"전세 사기 피해자 우선 선발" - 복지뉴스`, `복지서비스 상세(지자체) - 청년월세 지원사업`

“월세”처럼 매우 짧은 질의에 대해 검색 정확도 경고를 제공했고, 내부 VectorDB 검색 결과가 없어 Tavily 공식 도메인 검색으로 분기했다. 외부 검색 결과 3건을 기반으로 월세 관련 확인 후보를 제공했으므로 fallback 효과가 잘 드러난다. 다만 외부 결과는 상세 조건이 구조화되어 있지 않아 `확인 후보` 톤으로 답변한 점이 적절하다.

### TC010_no_empty_query

- 질문: ``
- Route: `-`
- Next Action: `-`
- External Used: `None`
- 추천 수: `0`
- 주요 추천/후보: 없음

빈 문자열 입력은 FastAPI/Pydantic 검증 단계에서 HTTP 422로 처리되었다. 평가 스크립트는 이를 서비스 장애가 아니라 입력 방어 로직으로 해석하여, 빈 질문 안내 메시지와 recommendations 빈 배열을 확인하고 통과 처리했다.

## 5. 평가 결과의 의미

이번 평가 결과는 시스템이 “정해진 실패를 피하고, 필요한 경우 외부 검색 fallback으로 분기하는 능력”을 상당히 안정적으로 갖추었음을 보여준다. 특히 과거/마감 공고 배제, 비현실적 질문에서 추천을 비우는 방어 로직, 짧은 질문에서 외부 검색으로 보강하는 흐름이 확인되었다.

반면, 의미적 추천 품질은 규칙 기반 점수만으로 완전히 보장되지 않는다. 자동 평가에서 100% 통과했더라도, 일부 케이스에서는 추천 후보가 질문과 직접 관련성이 낮거나, 가장 적절한 후보가 1순위에 오르지 않는 현상이 확인되었다. 따라서 최종 보고서에서는 다음과 같이 해석하는 것이 적절하다.

```text
규칙 기반 안정성: 매우 높음
ReAct 분기 및 fallback: 정상 작동
마감/오래된 정책 방지: 대체로 정상 작동
의미적 추천 품질: 일부 케이스에서 추가 개선 필요
BLEU/ROUGE: 추천형 RAG 특성상 낮게 나오므로 참고 지표로만 활용
```

## 6. 한계 및 추가 개선 방향

### 6.1 평가셋/평가 스크립트 개선

- `면접수당`처럼 route는 맞지만 실제 후보가 부적절한 경우를 잡기 위해 `expected_primary_policy_any`, `expected_primary_keyword_any` 같은 조건을 추가한다.
- `접수 마감`이 제목에 포함된 후보는 `deadline_status`와 별개로 감점 또는 제외하도록 평가한다.
- recommendations의 1순위 후보가 질문 핵심어를 얼마나 충족하는지 별도 점수화한다.
- LLM-as-a-Judge 결과를 정량 보고서에 함께 포함해 의미적 품질 평가를 보강한다.
- 규칙 기반 평가셋과 LLM Judge 평가셋의 필드 구조와 목적이 다르다는 점을 문서에 명확히 구분한다.

### 6.2 RAG 검색/랭킹 개선

- 동일 교육과정의 회차 중복 제거
- 제목 기반 마감 감지 강화: `(접수 마감)`, `마감`, `종료` 등
- direct intent keyword를 더 강하게 반영: `면접수당`, `데이터 분석`, `참여위원회` 등
- 내부 VectorDB 후보와 Tavily 외부 후보를 통합 ranking하는 방식 고도화
- 외부 검색 결과의 상세 페이지 본문 파싱을 통해 신청 기간, 대상, 지역, 신청 URL을 구조화
- Graph DB 보조 검색 결과가 답변 후보로 사용될 경우 source_url, application_url, summary 부족 문제를 보완

## 7. 보고서용 요약 문장

```text
RAG 서비스 평가 결과, 총 10개의 테스트 케이스가 모두 통과하여 규칙 기반 평가 기준 100% 통과율을 달성하였다. 평가 스크립트는 route, next_action, recommendations 개수, 마감 공고 배제, 외부 검색 사용 여부, ReAct tool_trace 존재 여부 등을 기준으로 점수를 산정하였다. 이를 통해 내부 VectorDB 기반 검색, ReAct 기반 검색 충분성 판단, Tavily 공식 도메인 외부 검색 fallback, 입력 검증 로직이 정상적으로 작동함을 확인하였다.

다만 BLEU/ROUGE는 추천형 RAG 특성상 낮게 산출되었으며, 일부 케이스에서는 규칙 기반 평가는 통과했지만 의미적으로 아쉬운 후보가 추천되었다. 따라서 최종 성능은 규칙 기반 안정성과 함께 LLM-as-a-Judge 및 케이스별 정성 분석을 병행하여 해석하는 것이 적절하다.
```
